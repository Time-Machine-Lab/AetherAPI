## Context

当前已知故障都发生在后端业务委托执行之前，表现为框架层直接返回 `400`。已确认受影响的接口覆盖分类列表与详情、资产详情、当前用户 API Key 列表，而同类 Controller 方法签名模式在目录发现、调用日志等接口中也存在。与此同时，父级 Maven 编译配置当前没有显式开启 Java 参数名元数据输出，而多个 Controller 的 `@PathVariable` 与 `@RequestParam` 又依赖隐式参数名推断。

该变更属于横切性修复，因为它同时涉及 Controller 方法签名、构建配置、全局异常映射三部分。另一个需要修正的现状是：仓库根目录 `docs/api/` 已存在权威文档，但当前变更提案误把文档缺口定位到了 `aether-api-hub/docs/api/`。实施阶段必须以 repo 根目录 `docs/api/` 为准，更新已有文档并补齐缺失的单 Controller 文档。

## Goals / Non-Goals

**Goals:**
- 让受影响后端接口在打包产物中也能稳定完成参数绑定。
- 让分类、资产、当前用户 API Key，以及其他同模式接口在框架级绑定失败时返回各自接口族下的 `400` 错误码，而不是统一落到 `CATEGORY_CODE_INVALID`。
- 避免客户端看到编译器参数、反射元数据等内部诊断信息。
- 增加覆盖成功绑定与绑定失败映射的回归测试。
- 因为本次变更会影响客户端可见错误行为，在代码实现前先补齐或更新 repo 根目录 `docs/api/` 下对应 Controller 的权威 API 文档。

**Non-Goals:**
- 不修改任何前端请求代码。
- 不重设计接口路径、请求体结构或领域校验规则。
- 不引入数据库或 SQL 变更。
- 不把本次提案扩展成对全部适配层接口的一次性重构，但必须完成同类模式扫描并覆盖高风险接口。

## Decisions

### 1. 同时加固构建元数据与 Controller 显式绑定名

实现阶段应同时补齐 Maven 的 Java 参数名元数据输出配置，并把受影响 Controller 中依赖隐式参数名推断的绑定注解改为显式命名。

之所以不只依赖编译参数，是因为显式绑定名可以让 Controller 契约更直观，也能降低未来构建链变化带来的回归风险。之所以不只改注解，是因为编译参数补齐仍然能为其它框架绑定路径提供兜底，减少潜在同类问题。

### 2. 将框架级绑定失败与领域校验失败分开处理

实现阶段应为框架级请求绑定异常提供独立分类逻辑，而不是继续落到当前通用 `IllegalArgumentException` fallback，并最终把无关接口都映射成 `CATEGORY_CODE_INVALID`。

之所以不保留现有通用 fallback，是因为它已经把资产与凭证接口错误误标成分类错误。之所以不引入全新的跨领域通用错误码，是因为当前接口族已经存在各自的无效请求语义，本次提案应优先采用最小的客户端可见修正。这里需要覆盖的不只是 `IllegalArgumentException`，还包括 Spring Web 的缺参、类型不匹配、路径变量绑定失败等框架异常。

### 3. 先补 Controller 级 API 权威文档，再进入代码实现

由于本次变更会修改客户端可见的错误行为，实现阶段应先使用 `tml-docs-spec-generate` 更新或补齐 repo 根目录 `docs/api/` 下的 Controller 级权威文档。

之所以不把文档放到代码之后，是因为当前 OpenSpec 规则要求任何影响接口行为的变更都要先落 API 权威文档。本次变更不涉及存储结构变化，因此不需要新增 `docs/sql/` 工作项。已有的 `api-credential.yaml`、`api-asset-management.yaml`、`api-category-lifecycle.yaml` 应优先评估复用或重整，而不是平行创建第二套冲突文档。

## Risks / Trade-offs

- [遗漏同类 Controller 模式] -> 通过全仓搜索建立受影响清单，并为提案范围内每类接口补充回归测试。
- [双重加固看起来比问题本身更宽] -> 代码改动仍限制在已知故障接口族，但共享构建兜底配置需要一并补齐，以消除根因依赖。
- [错误码按接口族映射可能变脆] -> 为分类、资产、凭证、目录发现、调用日志等接口族分别增加测试，避免继续依赖过于宽泛的 fallback。
- [文档重整范围失控] -> 只更新受本次变更影响的现有权威文档，缺失的单 Controller 文档再补齐，不顺手做全量文档重构。

## Migration Plan

1. 先为受影响 Controller 在 repo 根目录 `docs/api/` 下更新现有权威文档，并补齐缺失的单 Controller API 文档。
2. 增加能够复现当前失败现象的回归测试，并明确目标错误映射。
3. 只在后端实现编译参数、Controller 绑定、异常处理相关修改。
4. 通过定向测试和打包后烟雾验证确认问题消失。
5. 如需回滚，直接回退后端版本即可；本次不涉及数据库回滚。

## Open Questions

- 实施范围是否先覆盖当前已确认失败的分类、资产、凭证接口，并顺手修复目录发现与调用日志这类同模式高风险 Controller。
- 对于未来仍无法归类到具体接口族的框架级绑定失败，是否后续再单独提出一个更通用的 Web adapter 错误码设计变更。
