## Context

架构设计文档把“API 市场浏览与接入”列为第一阶段核心场景之一，并要求 API 市场页面从 Catalog 主数据中读取列表和详情。当前仓库没有任何 Catalog 查询接口，因此这项 change 需要建立一个尽量独立的读模型能力，让前端和后续集成方可以只依赖已启用资产进行浏览，而不直接耦合资产写模型实现细节。

该 change 主要影响 `api`、`adapter`、`service`、`infrastructure`，并依赖分类与资产主数据已经存在。实现时仍需遵守后端开发规范，避免在 Controller 里直接查库，避免把持久化对象直接暴露成对外契约。

## Goals / Non-Goals

**Goals:**

- 提供面向 API 市场和内部开发者浏览场景的已启用资产列表与详情查询。
- 明确普通 API 与 AI API 在详情输出中的差异化表达边界。
- 保持查询能力对写模型最小耦合，便于和资产生命周期并行推进。
- 明确后续需要补齐的 `docs/api/` 与 `docs/sql/` 权威文档，并要求后续使用 `tml-docs-spec-generate` 生成。

**Non-Goals:**

- 不在本 change 内实现资产注册、编辑、启停等写入行为。
- 不在本 change 内实现 Consumer/Auth、Unified Access 或 Observability。
- 不在本 change 内引入复杂搜索、排序治理、推荐或标签筛选。

## Decisions

### 1. 使用独立查询服务或读模型，而不是直接复用写聚合返回

查询能力以 `service` 层查询用例和 `infrastructure` 层查询实现为主，输出面向列表/详情场景的只读结果模型。这样可以减少对 `ApiAssetAggregate` 内部结构的耦合，也能避免未来资产写模型演进时频繁改动对外查询契约。

备选方案是让 Controller 直接查询仓储并暴露领域对象，但这既破坏分层，也会让读写模型互相牵连，因此不采用。

### 2. 公开浏览范围只包含已启用资产

领域设计文档要求“列表只展示处于启用状态的资产”，架构设计也把 Catalog 作为市场展示主数据。因此查询 API 只返回已启用资产，草稿和停用资产不应通过该读模型对外暴露。

备选方案是让查询接口返回全部状态，再由调用方筛选，但这会把领域规则外推给上层系统，因此不采用。

### 3. 详情视图按资产类型分层输出

普通 API 与 AI API 共用基础展示信息，但 AI API 详情需要额外返回供应商、模型、流式能力和能力标签等信息。读模型会把这部分作为 AI 扩展视图，而不是直接暴露内部持久化结构。

备选方案是返回一个完全松散的元数据 blob，由前端自行解释，但这会弱化契约可读性，也不利于后续 SDK 和市场页面稳定演进，因此不采用。

### 4. 第一阶段只做最小列表与详情，不提前塞入搜索和分类筛选

虽然架构文档提到分类浏览，但目前领域设计文档在用户故事里明确的最小闭环是“已启用资产列表和详情”。为了让该提案与分类管理、资产写模型保持最低耦合，第一阶段只做列表和详情，不把搜索、复杂筛选、排序治理塞进同一 change。

备选方案是一次性实现市场搜索和分类浏览，但会显著扩大 scope，也会增加与其他 change 的代码交叉，因此不采用。

### 5. 顶层 API/SQL 文档仍然先于代码落地

即使查询 change 以读模型为主，只要新增查询 API 或查询存储结构，就必须把 `docs/api/` 与 `docs/sql/` 视为权威设计产物。后续实现时应先使用 `tml-docs-spec-generate` 生成 Catalog 查询文档，再据此定义对外契约和查询实现。

## Risks / Trade-offs

- [Risk] 如果资产写模型尚未完成，查询 change 只能先依赖约定中的主数据结构。 → Mitigation：在 spec 中只约束外部可见行为，把内部表结构细节留给顶层 SQL 文档。
- [Risk] 只做最小列表和详情可能不足以支撑完整市场体验。 → Mitigation：明确把搜索、筛选、排序放到后续 change，而不是在本 change 内膨胀范围。
- [Risk] 查询接口若误暴露草稿、停用资产或敏感上游配置，会破坏领域边界。 → Mitigation：把“仅启用资产可见”和“输出面向展示而非透出内部配置细节”写入 spec。

## Migration Plan

1. 先生成并评审 Catalog 查询相关的 `docs/api/` 与 `docs/sql/` 顶层文档。
2. 在 `service` 中定义列表与详情查询用例，在 `infrastructure` 中实现对应查询模型。
3. 在 `api` 与 `adapter` 中补齐只读查询 DTO 与接口适配。
4. 增加接口测试，验证启用状态过滤与 AI 详情差异化输出。

## Open Questions

- 第一阶段列表接口是否需要分页参数；架构文档没有固定这一点。
- 查询接口是面向匿名访问、内部登录态，还是先面向内部网络开放；文档当前只定义了浏览场景，未固定访问控制方式。
- 详情页中的示例快照是否需要完整返回，还是只返回摘要并在前端做折叠展示。
