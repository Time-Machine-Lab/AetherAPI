## Context

`Observability` 是 Aether API Hub 一期剩余的必要领域之一，但本期目标非常克制，只要求平台能够把一次 API 调用的核心事实稳定记录下来，为开发者控制台的最小日志查看提供数据来源。现有顶层文档已经明确该领域不承担告警平台、监控大盘、计费结算或完整请求响应归档，因此本次设计必须围绕“最小调用日志基础能力”展开，并遵守现有 DDD 分层与模块边界。

当前仓库还没有与调用日志对应的顶层表结构文档，因此一旦进入实现，必须先通过 `tml-docs-spec-generate` 使用 SQL 模板生成或更新 `docs/sql/api_call_log.sql`，再开展 repository 与持久化实现。该 change 不向前端直接暴露接口，也不修改现有 `docs/api/` 契约。

## Goals / Non-Goals

**Goals:**

- 定义最小 API 调用日志写入模型，覆盖调用主体、目标 API、调用时间、耗时、结果分类、错误摘要与 AI 扩展预留字段。
- 明确 Observability 写侧只负责“记录平台事实”，不承担前端查询接口。
- 约束日志写入发生在 Unified Access 完成一次调用判定之后，避免在 adapter 层直接拼装和落库。
- 先生成 `docs/sql/api_call_log.sql` 作为顶层权威结构文档，再实现 domain / application / infrastructure 代码。

**Non-Goals:**

- 不建设日志查询接口、报表接口、统计接口与告警接口。
- 不记录完整原始请求体、响应体与大对象内容，避免一期引入存储膨胀与敏感信息风险。
- 不引入新的 Consumer 显式业务概念，也不变更 Consumer/Auth 既有边界。
- 不新建多张观测明细表、聚合表或离线分析表。

## Decisions

### 1. 以单表 `api_call_log` 承载一期最小调用事实

一期只需要稳定记录调用事实，因此使用单表 `api_call_log` 承载最小日志模型，而不是同时拆分主表、错误表、AI 扩展表。这样可以降低建模和联调复杂度，也更符合用户提出的“先做最小 API 日志查看”的目标。

备选方案是拆成多表，分别管理基础日志、错误详情和 AI 扩展，但这会让一期 SQL 文档、仓储实现和后续查询接口同时复杂化，且当前并无足够业务收益，因此不采用。

### 2. 日志写入由 Observability 应用服务触发，Unified Access 仅传递调用结果上下文

业务规则不能写入 Controller、Mapper 或第三方适配器，因此调用日志写入应由 Observability 应用服务承接，由 Unified Access 在调用完成后传入标准化调用结果上下文。这样可以保持跨领域协作清晰，也避免后续更换转发实现时影响日志模型。

备选方案是在 Unified Access adapter 或 infrastructure 中直接拼接日志实体并落库，但这会破坏分层边界，并把日志规则写死在接入或转发实现里，因此不采用。

### 3. 一期只记录平台侧可判定的摘要字段，不保留完整上下游报文

为了兼顾可查性、成本与安全性，日志表只保留调用请求方法、目标路径快照、状态码/结果分类、错误码/错误摘要、耗时等摘要信息，不在一期保存完整 header、body、response payload。这样既能满足开发者控制台排查基本问题，也避免引入敏感数据治理负担。

备选方案是直接完整落请求体与响应体，但这会迅速扩大存储规模，并增加脱敏与权限控制复杂度，不符合当前最小交付目标，因此不采用。

### 4. 为 AI 场景预留可空扩展字段，但不提前做独立能力建模

顶层领域文档已经指出后续可能需要 AI 调用观测，因此在 `api_call_log` 中保留 AI 模型标识、token 摘要或扩展元数据等可空字段是合理的。但这些字段在一期只作为预留，不构成独立业务流程，也不新增单独接口或单独聚合。

备选方案是完全不预留 AI 相关字段，等后续再改表；这会导致后续领域演进需要额外迁移成本，因此不采用。

## Risks / Trade-offs

- [Risk] `api_call_log` 字段过多会让“一期最小化”失焦。 → Mitigation：字段仅覆盖控制台日志查看所需最小事实，其余复杂观测数据留到后续 change。
- [Risk] Unified Access 输出的调用结果分类如果不稳定，会导致日志写侧难以沉淀统一模型。 → Mitigation：在本 change 中仅依赖统一的结果快照输入，不反向侵入 Unified Access 的主业务规则。
- [Risk] 不保存完整报文可能影响部分深度排障。 → Mitigation：一期先满足“日志可查”，后续如有明确需求再通过独立 change 引入受控的原文观测能力。

## Migration Plan

1. 使用 `tml-docs-spec-generate` 的 SQL 模板生成或更新 `docs/sql/api_call_log.sql`，明确表结构、主键、索引与字段语义。
2. 在 Observability 领域实现调用日志聚合、写入命令、仓储接口与持久化映射。
3. 在 Unified Access 的调用完成边界接入 Observability 应用服务，传递标准化调用结果上下文。
4. 增加成功、失败、平台前置失败、AI 扩展预留字段等核心测试。

## Open Questions

- `api_call_log` 是否需要在一期保存请求来源 IP 或 User-Agent 摘要；当前建议仅在不引入额外敏感治理负担时再纳入。
- AI 预留字段最终采用独立列还是单个扩展 JSON 字段；当前建议优先保留少量明确字段，避免过早泛化。
