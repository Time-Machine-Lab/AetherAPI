## Context

`Consumer & Auth` 的前一条主链路是“用户拿到并管理自己的 API Key”，后一条主链路则是 `Unified Access` 在收到请求后，能够稳定识别调用方并返回后续模块可消费的 `Consumer Context`。架构文档已经明确 `Unified Access -> Consumer & Auth -> Observability` 的调用顺序，因此本 change 需要落地统一接入侧的鉴权校验和上下文组装，但不能因此倒逼出新的显式 Consumer 业务接口。

与 `consumer-auth-key-management` 相比，这个 change 更偏向后端内部能力：它需要复用已有的凭证主数据和隐式 Consumer 映射，提供统一的验证结果与失败原因，并在校验后更新最近使用快照，为日志、控制台与后续治理能力提供稳定基础。

## Goals / Non-Goals

**Goals:**

- 为统一接入提供进程内可调用的凭证校验能力，返回结构化 `Consumer Context`。
- 将 API Key 指纹匹配、凭证状态判断、过期判断和 Consumer 可用性判断收敛在应用服务和领域对象内。
- 为调用日志和控制台预留最近使用快照更新行为，但不承担日志查询本身。
- 保持实现方式为模块内应用服务或端口调用，不额外引入用户可见 HTTP 鉴权接口。

**Non-Goals:**

- 不实现 API Key 创建、列表、详情、停用、启用、吊销等用户侧管理接口。
- 不实现显式 Consumer 管理接口或页面。
- 不实现调用日志查询、配额、限流、计费和安全治理策略。
- 不新增独立 SQL 文档；若需要的字段已由 `consumer-auth-key-management` 覆盖，则直接复用已有权威设计。

## Decisions

### 1. 统一接入通过应用服务调用鉴权能力，而不是新增独立 HTTP 鉴权接口

在模块化单体边界内，`Unified Access` 应优先通过 `service/api` 层定义的应用服务或端口调用 `Consumer & Auth` 能力，拿到结构化 `Consumer Context`。这样可以避免在当前阶段为了内部协作再长出一套对内 HTTP 协议和控制器。

备选方案是增加一个独立的内部鉴权 Controller 或 HTTP Endpoint，但这会引入额外的序列化层和文档负担，也不符合当前“避免多余接口设计”的要求，因此不采用。

### 2. 鉴权结果必须返回结构化上下文与分类失败原因

鉴权能力不是简单返回 `true/false`，而是需要返回 `ConsumerId / ConsumerCode / CredentialId / CredentialStatus` 等上下文，以及可区分的失败原因类型，例如“凭证不存在”“凭证停用”“凭证吊销”“凭证过期”“Consumer 不可用”。这既方便统一接入决定后续行为，也为日志和错误码对齐提供基础。

备选方案是只返回布尔值和字符串消息，但这会让后续模块继续自行拼装上下文或自行判断失败语义，导致规则散落，因此不采用。

### 3. 最近使用快照在鉴权链路内统一更新

当请求能够匹配到某个已存在的凭证时，鉴权能力负责在同一链路内更新该凭证的最近使用时间与结果快照；不存在的凭证由于没有归属对象，不更新任何快照。这样控制台查询、日志补充和未来治理规则都能依赖同一来源。

备选方案是把最近使用更新丢给日志模块或统一接入调用方自行维护，但这会造成多处更新同一字段，且在失败场景下容易遗漏，因此不采用。

### 4. 存储结构复用前置提案，不在当前 change 重复定义

本 change 默认复用 `consumer-auth-key-management` 中定义的凭证表、Consumer 映射表和最近使用字段，不单独发明新的表结构或第二份 SQL 文档。若实现过程中发现前置 SQL 缺少必要字段，应通过调整前置提案补齐，而不是在当前 change 内再引入平行存储设计。

备选方案是当前 change 再补一份独立 SQL 文档，但那会直接与凭证管理 change 在顶层文档和代码上产生交叉修改，因此不采用。

## Risks / Trade-offs

- [Risk] 该 change 依赖前置凭证主数据，如果前置提案尚未定稿，当前设计容易悬空。→ Mitigation：在任务中显式依赖前置 SQL/领域模型，必要时先同步补齐。
- [Risk] 鉴权失败原因过细会让错误码设计变复杂。→ Mitigation：先按领域语义定义稳定分类，具体错误码映射在实现阶段与现有集中错误码方案对齐。
- [Risk] 最近使用快照更新如果放在高频链路中，后续可能带来写放大。→ Mitigation：一期先保证语义正确，若后续性能成为问题，再通过异步化或批量化方案优化。

## Migration Plan

1. 确认并复用 `consumer-auth-key-management` 已批准的 SQL 文档与凭证主数据模型。
2. 在 `domain/service` 中实现凭证校验、Consumer 状态判断和 `Consumer Context` 组装能力。
3. 在 `infrastructure` 中补齐按指纹查询凭证、加载 Consumer 和更新最近使用快照的仓储实现。
4. 在 `Unified Access` 使用点接入该应用服务，并将结构化失败原因映射为统一接入层可处理的错误结果。
5. 增加应用服务测试和集成测试，覆盖凭证不存在、停用、吊销、过期、Consumer 不可用以及成功校验场景。

## Open Questions

- 一期是否需要为“匹配到已停用/吊销/过期凭证”的失败场景写入最近失败时间；当前设计建议写入，因为这对安全治理更有价值。
- 如果未来 `Unified Access` 被拆为独立服务，是否仍保持进程内调用；当前设计默认模块内调用，若架构演进再通过新的 change 外置接口。
