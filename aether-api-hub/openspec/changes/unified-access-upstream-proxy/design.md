## Context

`Unified Access` 的统一入口、目标匹配和前置校验只是打通主链路的第一半，真正让平台“可统一调用”的关键，还在于把请求稳定地送到真实上游，并把成功结果按原始语义返回给调用方。领域文档已经明确：

- 成功调用场景不应使用 `TML-SDK Result`
- 统一接入层应尽量保留上游状态码、响应头和响应体语义
- AI API 在支持流式时应保留流式透传边界

因此这份 change 专注于真实上游转发、成功返回与流式边界，不再重复入口契约和前置失败设计。

## Goals / Non-Goals

**Goals:**

- 实现统一接入层的上游请求组装与真实转发执行。
- 保证成功调用时优先保留上游状态码、响应头和响应体语义，而不是包装成 `TML-SDK Result`。
- 为 AI API 的流式透传保留最小可用边界，不阻断后续流式输出。
- 复用 `unified-access-entry-routing` 提供的统一入口和目标快照，不再额外发明第二套入口契约。

**Non-Goals:**

- 不重新定义统一入口路径、Controller 契约和平台前置失败分类。
- 不新增新的顶层 SQL 文档。
- 不实现完整 AI Gateway、多模型路由、Fallback、限流或熔断能力。
- 不实现日志查询和控制台展示接口。

## Decisions

### 1. 成功响应优先保留上游原始语义

统一接入层在成功调用场景下应尽量回传上游的原始状态码、关键响应头和响应体，而不是包装成平台内部业务接口所使用的 `TML-SDK Result`。这样可以最大程度保留第三方 API / AI Provider 的真实行为，也符合领域文档已经明确的边界。

备选方案是把所有成功响应统一转换成平台返回包装，但那会破坏普通 API 的原始语义，也会明显影响 AI API 的流式响应处理，因此不采用。

### 2. 请求组装采取“最小必要处理”原则

上游请求组装只做以下必要动作：

- 透传 query 参数
- 透传 body
- 透传允许继续传递的 headers
- 注入平台到上游所需的鉴权信息
- 清理不应继续向上游暴露的内部头信息

不在一期引入复杂参数改写、脚本化请求转换或多上游路由规则。这样既符合 `Gateway-light`，也让实现边界更清晰。

备选方案是引入更强的请求映射能力，但这会迅速把轻量统一接入拉向重网关实现，因此不采用。

### 3. 流式能力按“保边界优先”实现

对于 `API Catalog` 已标记为支持流式的目标 API，本提案要求转发层保持流式透传边界。即便一期只打通最小样例，也不能在架构设计上把流式响应强制折叠为普通响应。

备选方案是一期全部转换为普通同步响应，但这会与 AI-ready 方向冲突，也会让后续回补流式成本显著增加，因此不采用。

### 4. 上游执行器与入口控制器保持分离

真实转发执行应落在独立的应用服务或基础设施执行器中，由统一入口调用，而不是把 HTTP 客户端、头处理和透传逻辑全部塞进 Controller。这样更符合 DDD 分层，也更适合后续替换具体 HTTP 调用实现。

备选方案是由入口控制器直接构造并发起 HTTP 请求，但这会让 adapter 层承担太多业务规则和执行责任，因此不采用。

## Risks / Trade-offs

- [Risk] 上游成功响应原样回传后，不同 API 的返回风格会很不统一。→ Mitigation：接受这种差异，因为统一接入层的职责是“统一入口”，不是“统一业务结果结构”。
- [Risk] 流式透传在一期实现难度高于普通同步调用。→ Mitigation：先要求设计和能力边界明确，至少打通一个最小 AI API 样例。
- [Risk] 如果入口提案和转发提案并行推进不当，容易出现接口边界对不上。→ Mitigation：本提案明确依赖 `unified-access-entry-routing` 输出的目标快照和调用结果边界。

## Migration Plan

1. 复用 `unified-access-entry-routing` 确认后的统一入口和目标快照模型。
2. 在 `service` 中定义上游转发应用服务边界，接收已解析的 `Consumer Context` 与 `TargetApiSnapshot`。
3. 在 `infrastructure` 中实现 HTTP 上游调用执行器、头处理和响应透传逻辑。
4. 补齐普通调用成功、上游失败和最小流式样例的测试。
5. 将调用执行结果暴露给后续 `Observability` 记录使用。

## Open Questions

- 一期最小流式样例选哪个 AI API 来验证；当前 design 保留为实施前的样例 API 选择问题。
- 是否需要保留一份“允许透传的请求头白名单”；当前建议保留，但具体名单应在 API/实现层再定。
