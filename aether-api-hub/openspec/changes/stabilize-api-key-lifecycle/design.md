## Context

联调记录显示 API Key 创建成功，说明当前用户到隐式 Consumer 的映射、凭证签发和首次持久化基本可用；但停用和吊销返回默认 `500`，说明问题集中在已存在凭证的状态变更、聚合重建或持久化更新路径。

当前代码中 API Key 生命周期由 `ApiCredentialAggregate` 维护状态规则，应用服务通过 `loadCredential(currentUserId, credentialId)` 保证当前用户边界，再调用仓储保存。因此修复应保持现有 DDD 分层，不应在 controller 中绕过业务规则。

## Goals / Non-Goals

**Goals:**

- `disable`、`enable`、`revoke` 成功时返回更新后的 API Key 响应。
- 重复停用、重复吊销、吊销后启用等状态冲突返回明确业务错误。
- 当前用户无法操作不存在或不属于自己的凭证。
- 联调测试数据可通过公开接口停用或吊销清理。

**Non-Goals:**

- 不新增 API Key 删除接口。
- 不暴露 Consumer 注册、启用、停用等显式接口。
- 不改变 API Key 明文只返回一次的安全语义。

## Decisions

### 1. 修复状态变更写路径，而不是新增清理接口

- 决策：继续使用现有 `disable` / `revoke` 接口完成测试数据清理。
- 原因：契约已定义这些生命周期动作，当前问题是实现稳定性不足。
- 备选方案：新增测试清理接口或删除接口。
- 不采用原因：会扩大业务范围，也可能破坏凭证审计语义。

### 2. 保持 API Key 状态规则在聚合内

- 决策：状态可变更性仍由 `ApiCredentialAggregate` 判断，应用服务只负责加载、调用和保存。
- 原因：避免 controller、delegate、repository 分散业务规则。
- 备选方案：在 SQL 更新时直接改状态。
- 不采用原因：会绕过领域规则，容易产生吊销后恢复等安全问题。

### 3. 持久化更新必须验证真实落库

- 决策：测试不仅断言返回体，还要重新查询列表/详情验证状态已经落库。
- 原因：当前问题发生在更新路径，单纯内存模型返回不能证明数据库状态正确。
- 备选方案：只测 application service 返回值。
- 不采用原因：无法发现 MyBatis 更新、乐观锁或字段转换问题。

## Risks / Trade-offs

- [乐观锁版本处理可能导致更新未命中] -> 实现时需要确认 MyBatis-Plus `@Version` 的旧值和新值处理方式，并通过查询验证。
- [状态错误映射依赖异常 message] -> 如有必要可引入更明确的领域错误类型，但本期优先保持小改动。
- [残留测试 Key 已存在异常状态] -> 修复后需要用公开接口复测新建 Key，并视情况处理历史残留数据。

## Migration Plan

1. 复现 `disable` / `revoke` 默认 `500` 并记录异常栈。
2. 修复状态变更保存路径。
3. 补充生命周期状态冲突测试。
4. 使用新建 API Key 完成创建、停用、启用、吊销、列表状态筛选复测。
