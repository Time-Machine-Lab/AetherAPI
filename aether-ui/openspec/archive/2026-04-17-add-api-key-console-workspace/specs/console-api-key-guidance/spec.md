## ADDED Requirements

### Requirement: 控制台 SHALL 解释一次性密钥展示和凭证安全规则

凭证体验 SHALL 包含可见说明，明确 API Key 明文只会在创建成功后展示一次，而控制台后续界面只能提供掩码展示。

#### Scenario: 展示创建后的明文提示

- **WHEN** 创建流程返回一个新生成的 API Key
- **THEN** 控制台展示一个专门承载“一次性明文密钥”的成功态
- **THEN** 同一状态中包含安全提示，告知用户需要立即保存该 key，因为后续界面不会再次展示完整密钥

#### Scenario: 创建完成后再次进入凭证工作区

- **WHEN** 用户在初次创建成功态消失后重新回到工作区
- **THEN** 控制台只展示掩码凭证数据
- **THEN** 控制台继续保留提醒文案，说明完整密钥无法再从 UI 中恢复

### Requirement: 控制台 SHALL 提供边界清晰的凭证使用说明

控制台 SHALL 提供关于 API Keys 如何融入“当前用户接入模型”的说明，并且这些说明 SHALL 严格受现有 Consumer & Auth 权威文档边界约束。

#### Scenario: 解释隐藏的 Consumer 模型

- **WHEN** 工作区为用户渲染说明内容
- **THEN** 控制台解释 API Keys 是按当前登录用户维度进行管理的
- **THEN** 控制台不会引导用户手动创建或管理独立的 `Consumer`

#### Scenario: 缺少可执行调用示例的权威文档

- **WHEN** 前端缺少某个具体统一接入请求示例的已批准权威文档
- **THEN** 控制台展示边界清晰的说明文案，或引导用户前往现有 docs 区域
- **THEN** 控制台 MUST NOT 把未文档化的请求头、接口路径或请求体伪装成正式契约进行展示

### Requirement: 控制台 SHALL 对所有用户可见的凭证说明和错误进行国际化

所有用户可见的凭证说明、空状态、操作反馈和可恢复错误提示 SHALL 通过控制台现有 i18n 结构提供，而不是写成硬编码页面字符串。

#### Scenario: 在不同语言环境下渲染说明内容

- **WHEN** 用户在 `zh-CN` 和 `en-US` 之间切换控制台语言
- **THEN** 凭证说明、状态文案和面向用户的错误提示会随当前语言一起切换
- **THEN** 工作区不会把新增的凭证相关字符串遗漏在 i18n 资源结构之外
