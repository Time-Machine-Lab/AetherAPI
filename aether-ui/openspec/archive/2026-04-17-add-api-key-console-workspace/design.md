## 背景

`aether-console` 已经是 API Hub 市场浏览和受保护工作区操作的统一前端入口，侧边栏里也预留了 `credentials` 菜单，但它目前还没有真实功能支撑。与此同时，后端权威文档已经给出了一期清晰规则：用户管理的是自己的 API Key，`Consumer` 仍然是内部模型，API Key 明文只展示一次，而 `lastUsedSnapshot` 由统一接入侧维护，而不是由页面侧自行生成。

因此，前端需要一份设计方案，在不突破 `docs/api/api-credential.yaml` 当前契约边界的前提下，适配现有控制台壳层、基于 `axios` 的 API 层、i18n 规则以及 `aether-console/DESIGN.md` 的语义约束。方案不能凭空补出未文档化的统一接入请求头、隐藏存储字段，或单独设计一套 `Consumer` 管理 UI。

## 目标 / 非目标

**目标：**

- 在 `aether-console` 中增加一个凭证管理工作区，让当前用户可以基于 `docs/api/api-credential.yaml` 创建、查看和操作 API Keys。
- 以对开发者友好的方式展示凭证生命周期信息，包括掩码 key、过期时间、状态和 `lastUsedSnapshot`。
- 提供“明文仅展示一次”的交互流程和持续性的安全提示，让后端的一次性密钥规则对用户足够明确。
- 复用现有控制台壳层、语义角色、API 层、路由 hash 导航和 i18n 结构，不新增第二个应用，也不再并行做一套管理后台体验。
- 通过间接表达而不是显式管理的方式，让前端体验和“隐藏 `Consumer`”的产品模型保持一致。

**非目标：**

- 不创建任何显式的 `Consumer` 页面、路由、面向控制器的概念或用户可编辑的 `Consumer` 工作流。
- 不在本次变更中补 usage、billing、orders、API 调用日志或 observability 页面，这些当前都还是占位导航项。
- 不修改 `docs/api/api-credential.yaml`，不发明新的统一接入 HTTP 调用契约，也不在本 change 中新增后端接口。
- 不为凭证编排新增全局业务 Store；异步编排继续放在 `api` 模块和页面级组合逻辑中。

## 关键决策

### 1. 将新体验落在现有 `console-workspace` 的凭证锚点下

- 决策：第一版 API Key 管理能力放在现有 `console-workspace` 路由和 `#credentials` 锚点下实现，而不是新建单独路由树或第二个控制台应用。
- 原因：`console-shell.ts` 已经将 `credentials` 建模为运营侧工作区中的一个分区，而当前产品结构本身也是把市场浏览和受保护操作统一放进同一个 `aether-console` 壳层中。
- 备选方案：新建独立的凭证路由族，或者把能力迁移到 `aether-admin-console`。
- 不采用原因：独立路由族会在能力尚未验证前就引入额外的导航和状态重复；`aether-admin-console` 也不符合当前目标产品方向。

### 2. 数据访问保持为独立凭证 API 模块加页面级组合逻辑

- 决策：在 `src/api` 下增加凭证领域 API 模块，并补充页面级的组合逻辑 / 辅助函数，用于过滤、刷新、操作态管理以及明文一次性展示处理。
- 原因：前端规范要求页面负责状态装配和交互编排，但不能直接发裸请求；Store 也应继续保留给认证这类真正全局共享状态。
- 备选方案：把所有请求逻辑直接写进工作区页面，或新增一个 Pinia 凭证 Store。
- 不采用原因：页面层裸请求违反现有分层规则，而全局 Store 又会把本地工作区状态过度中心化。

### 3. 工作区采用“列表 + 详情 + 创建后展示”拆分式界面

- 决策：工作区按“凭证列表 + 状态筛选 + 选中详情 + 创建后明文展示与安全提示”来组织。
- 原因：API 契约天然区分了列表、详情和生命周期操作，而当前控制台设计系统也已经支持 Surface 卡片、状态 Badge、通知条和选中详情面板。
- 备选方案：纯表格布局，或多页面向导式流程。
- 不采用原因：纯表格不利于清晰承载一次性明文展示和详情状态；多页面向导又会给当前壳层引入不必要的路由复杂度。

### 4. 状态和最近使用信息以后端字段为唯一事实来源

- 决策：凭证状态、过期、吊销和 `lastUsedSnapshot` 全部严格以 API 返回值为准，前端只做格式化和空状态呈现。
- 原因：`consumer-auth-unified-access-auth` 已明确最近使用快照由后端维护，前端不应该自行推断鉴权成功与否、拼装安全事件或伪造隐藏字段。
- 备选方案：在前端额外推导使用状态，或者在浏览器本地缓存一份伪使用历史。
- 不采用原因：那会制造第二份非权威审计信息，也会破坏“权威事实来自后端契约和领域文档”的约束。

### 5. 接入说明要明确，但必须受权威文档边界约束

- 决策：在工作区中加入凭证说明模块，解释隐藏 `Consumer` 的模型、一次性密钥处理规则，以及官方接入细节应去哪里查看；对于尚未文档化的统一接入可执行示例，明确不在前端中自行发明。
- 原因：开发者确实需要基本上手说明，但归档的 unified-access 后端提案也明确说明，一期不会新增面向用户的 HTTP 鉴权接口。
- 备选方案：直接生成包含假定请求头、路径和示例体的完整请求示例面板。
- 不采用原因：这会超出现有权威文档边界，并可能向用户传递一个后端尚未正式发布的伪契约。

## 风险 / 权衡

- [Risk] 后端交付可能落后于前端提案，尤其是 `lastUsedSnapshot` 更新或生命周期操作可用性。 -> Mitigation: spec 严格绑定 `docs/api/api-credential.yaml`，对于快照字段仅定义空态 / 不可用态，不发明兜底数据。
- [Risk] 用户可能误以为离开创建成功页后还能再次获取完整 API Key。 -> Mitigation: 把一次性明文展示做成独立成功态，并配合持续性警告文案，后续所有界面只显示 `maskedKey`。
- [Risk] 现有工作区已经承载了 catalog 管理，新增 credentials 可能让路由显得拥挤。 -> Mitigation: 凭证体验固定锚定在自己的分区中，并复用现有 sidebar hash 导航，而不是把操作混进 catalog 面板里。
- [Risk] 说明文案容易逐步漂移成未文档化的鉴权行为。 -> Mitigation: 说明内容只允许复用现有权威文档中的概念；任何新增示例或鉴权契约文字都必须先补顶层文档。

## 迁移计划

1. 在开始实现前，先确认 `docs/api/api-credential.yaml`、Consumer & Auth 领域文档以及 `aether-console/DESIGN.md` 仍然是当前权威来源。
2. 在 `aether-console` 中补齐凭证 API 模块、DTO 映射和页面级编排逻辑，保证页面层不出现裸请求。
3. 落地凭证工作区分区、创建 / 明文展示流程、生命周期操作和 i18n 文案。
4. 补齐说明文案，以及最近使用快照不可用时的 empty / loading / error 状态。
5. 在允许继续扩展 usage 或 observability 等后续能力前，通过测试、`lint`、`type-check` 和 `build` 完成工作区验证。
