## 背景

`aether-console` 已经具备符合 API Hub 场景的基础骨架：公开可访问的首页、受保护的工作台页面、统一 `axios` 实例、路由守卫、国际化和 `ConsoleLayout`。与此同时，`docs/design/aehter-api-hub/` 与 `docs/api/` 已经为 API 市场浏览、资产管理和分类管理定义了清晰的业务边界与接口契约。

当前缺口不在后端契约，而在前端产品面还停留在模型市场骨架示例，尚未承接 `API Catalog Discovery`、`API Asset Management` 和 `Category Lifecycle` 三组接口。另一个关键约束是：现有资产管理契约只提供“创建 / 查询单个资产 / 修订 / 启停 / 绑定 AI 档案”，并未提供资产列表接口，因此管理端不能伪造一个完整的远程资产表。

## 目标 / 非目标

**目标：**

- 将 `aether-console` 首页改造为公开 API 市场页面，展示已启用资产列表、详情与 AI 能力信息。
- 将 `console-workspace` 改造为受保护的 API Catalog 工作台，覆盖分类管理和按 `apiCode` 驱动的资产管理流程。
- 通过 `src/api`、`src/features` 和页面编排完成接口对接，保持页面层不直接编写裸请求。
- 将现有导航、标题和 i18n 文案收束到 API Hub 语义，同时保留现有 `ConsoleLayout` 视觉语言。

**非目标：**

- 不修改 `docs/api/*.yaml` 中的后端契约，不新增资产列表、删除或批量操作接口。
- 不改造 `aether-admin-console`，该应用已弃用。
- 不实现真实用户体系、真实权限模型或统一调用入口，只沿用现有演示认证壳层。
- 不引入新的全局 Store 来承载页面局部编排状态。

## 决策

### 1. 使用单应用承载“市场页 + 工作台”

- 决策：所有 API Catalog 前端能力都落在 `aether-console`。
- 原因：架构文档定义的产品面就是“API 市场页面 + 开发者控制台”，而 `aether-console` 已天然具备公开首页和受保护工作台两种路由形态。
- 考虑过的备选方案：在 `aether-admin-console` 中实现管理界面。
- 不采用原因：该应用已经弃用，而且会偏离现有业务文档描述的单产品面结构。

### 2. 通过业务域 API 模块 + 页面本地编排完成对接

- 决策：在 `src/api/catalog/` 下新增 DTO、类型与 API 方法；在 `src/features/catalog/` 下承接页面需要的格式化、状态派生和展示辅助；页面只负责调用与编排。
- 原因：符合前端规范里“页面不直接写裸请求、Store 不堆积业务编排”的要求，也便于后续继续扩展 Consumer/Auth 与 Observability 相关页面。
- 考虑过的备选方案：将所有请求直接写在 `index.vue` / `workspace.vue` 中，或集中塞进现有认证 Store。
- 不采用原因：会破坏分层边界，并让后续测试与演进成本上升。

### 3. 市场页采用“列表 + 选中详情”的单页浏览模型

- 决策：`console-home` 首屏拉取 Discovery 列表，页面内部支持搜索、类型 / 分类筛选，并通过选中态展示详情面板；详情数据按需调用 `GET /api/v1/discovery/assets/{apiCode}`。
- 原因：现有 Discovery 契约明确支持列表与详情，两者都只暴露浏览字段，适合首页完成“发现 + 阅读”闭环。
- 考虑过的备选方案：仅展示列表而不请求详情；或者创建独立详情路由。
- 不采用原因：仅展示列表无法验证详情契约；独立详情路由会在当前 MVP 阶段增加额外路由与状态同步复杂度。

### 4. 工作台采用“分类面板 + 资产按编码管理”的模型

- 决策：`console-workspace` 拆成分类管理区与资产管理区。分类区使用分页列表接口；资产区通过 `apiCode` 查询单个资产、创建草稿并编辑当前资产，同时记录本地“最近操作资产”快捷入口。
- 原因：后端没有资产列表接口，按编码管理是唯一契约内可落地的真实工作流；本地最近记录能改善操作体验，但不会伪装成权威数据。
- 考虑过的备选方案：通过 Discovery 列表拼接成“资产管理列表”。
- 不采用原因：Discovery 只返回已启用资产，无法覆盖草稿与停用资产，会误导用户对数据完整性的认知。

### 5. 保留现有 ConsoleLayout，仅重写导航与文案语义

- 决策：继续使用现有 `ConsoleLayout` 和视觉 token，只把导航分组、顶部文案、首页结构和工作台模块换成 API Hub 语义。
- 原因：`aether-console/DESIGN.md` 已定义浏览型、卡片型、Airbnb 风格的视觉方向，适合 API 市场的“可发现”目标；复用布局还能减少无关改动。
- 考虑过的备选方案：重新设计一套更偏后台的数据密集布局。
- 不采用原因：会偏离当前项目级设计文档，也会放大本次需求的范围。

## 风险 / 权衡

- [资产管理没有远程列表接口]：工作台只能提供按 `apiCode` 查询和本地最近记录，页面中需要明确提示“最近操作”不是服务端全量清单。
- [首页详情采用二次请求]：在慢网络下会增加一次等待；通过保留列表摘要和详情加载占位降低感知中断。
- [现有演示认证不代表真实权限]：仅用路由守卫区分公开浏览和受保护工作台，不在页面内追加新的权限假设。
- [现有本地化文案与业务语义不匹配]：统一重写 `aether-console` 中英文地化资源，避免沿用模型市场骨架导致界面语义混乱。
