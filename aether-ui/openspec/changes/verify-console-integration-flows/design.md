## Context

当前 `aether-console` 已经具备真实联调所需的主要页面与接口编排入口：

- `src/pages/sign-in.vue`、`src/composables/useConsoleAuth.ts`、`src/stores/useAuthStore.ts` 与 `src/app/route-guards.ts` 共同承接控制台会话登录、恢复与受保护路由跳转；
- `src/pages/index.vue` 承接 API 市场浏览；
- `src/pages/workspace.vue` 承接分类管理、资产管理、API Key 与 API 调用日志工作台；
- `src/pages/playground.vue` 与 `src/features/unified-access` 承接统一接入调用工作台；
- `src/api/http.ts` 统一处理 `Authorization` 注入、错误归一化、401 会话失效回退与 mock adapter 分支。

同时，当前测试覆盖仍然偏窄：

- 现有自动化测试主要集中在 `app-config`、控制台导航结构、本地化资源和 API 调用日志 API 映射；
- 会话恢复、路由守卫、HTTP 401 失效处理、API Key 生命周期、Unified Access 关键交互、分类与资产管理等高风险链路缺少回归保障；
- 仓库虽然已具备 `Vitest` 基础，但尚未形成“自动化回归 + 真实后端冒烟 + 失败证据留存”的统一联调路径。

另外，`aether-ui/openspec/project.md` 当前不存在，因此本提案只能基于子项目 `openspec/config.yaml`、`aether-console/DESIGN.md`、根目录 API 契约文档和当前代码现状来收敛测试范围。

## Goals / Non-Goals

**Goals**

- 为 `aether-console` 建立一套可重复执行的联调验证基线，覆盖认证、市场、工作台、API Key、统一接入和调用日志等核心业务流。
- 明确“自动化契约回归”和“真实后端冒烟验证”的分层边界，避免把所有验证都压给人工，也避免在自动化不足时误判联调通过。
- 把真实联调所需的环境前置、mock 开关、测试账号、测试数据准备、失败证据记录和结果归档方式固化下来。
- 保证测试结论能够区分控制台会话失效、Unified Access API Key 失败、业务数据为空、接口契约不匹配等不同问题类别。

**Non-Goals**

- 不在本次提案中新增后端接口、修改数据库结构或扩展 `docs/api/*.yaml` 契约语义。
- 不把范围扩大到 `aether-admin-console` 或 `aether-web-marketing`。
- 不默认引入 Playwright、Cypress 等全新测试平台作为首选方案；首轮以现有工程能力和最小必要补强为主。
- 不把本次测试提案扩展为产品功能迭代，不新增页面或业务流程。

## Decisions

### 1. 以 `aether-console` 为唯一落地范围，围绕已对接的真实业务切片建立验证矩阵

- 决策：本次变更只覆盖 `aether-console`，并围绕已经存在真实前后端对接痕迹的业务切片建立测试矩阵：控制台认证、API 市场浏览、分类管理、资产管理、API Key 生命周期、Unified Access、API 调用日志。
- 原因：这些能力都已经在当前代码与根目录 API 契约中有明确映射，是“联调完成后必须验证”的最小闭环集合。
- 不采用“先做一个抽象测试框架、以后再填业务”的原因：会延后真正可执行的回归价值，也不符合当前用户要先落一份详细提案的目标。

### 2. 采用“自动化回归优先，真实后端冒烟补位”的双层策略，而不是默认引入新的端到端框架

- 决策：优先基于现有 `Vitest`、当前 `*.spec.ts` 组织方式和工程脚本补齐自动化回归；对 API Key 创建后一次性明文返回、Unified Access 真正触发上游、调用日志最终落库可查等依赖真实后端状态写入的链路，再补充人工冒烟验证。
- 原因：当前仓库已经具备 `Vitest` 运行基础，但没有现成的 Playwright/Cypress 组织与维护成本基线；先把可自动化的高风险链路覆盖起来，能更快形成稳定回归门槛。
- 取舍：这意味着首轮提案不会追求“一切都自动化”，而是明确保留一部分真实环境冒烟步骤和结果记录义务。

### 3. 把联调验证组织成“业务流闭环”，而不是零散的模块级测试清单

- 决策：任务与验证结果按业务流组织，而不是只按模块列测试文件。关键闭环至少包括：
  - 登录 -> 进入受保护页面 -> 刷新恢复 -> 退出/失效回退；
  - 市场浏览 -> 工作台管理 -> 返回确认；
  - 创建 API Key -> 使用 API Key 调用 Unified Access -> 在 API 调用日志中检索并查看详情。
- 原因：用户提出的是“接口与业务流程测试”，单纯的模块级 API 映射测试不足以证明跨页面、跨接口、跨状态链路已经闭环。

### 4. 真实后端联调必须显式确认非 mock 环境，并把环境前置作为通过条件的一部分

- 决策：在真实联调执行前，必须显式确认当前 `apiBaseUrl`、mock 开关状态、测试账号、基础数据和清理策略；如果验证实际仍命中本地 mock adapter 或演示数据，则该次结果不能视为联调通过。
- 原因：当前 `src/api/http.ts` 仍保留 mock adapter 分支，不把环境边界写清楚，容易出现“界面可操作但未连上真实后端”的假阳性结果。
- 取舍：这会增加一点执行前检查成本，但能显著降低联调结论失真风险。

### 5. 联调失败记录必须以稳定证据为中心，尤其要区分控制台会话失败与 Unified Access 业务失败

- 决策：对登录失败、401 会话失效、API Key 无效、Unified Access 平台前置失败、调用日志查询失败等场景，测试记录中至少保留失败步骤、入口页面、请求条件、稳定错误码或 `traceId`、环境信息与是否可复现。
- 原因：`CONSOLE_SESSION_UNAUTHORIZED` 与 Unified Access 的 `INVALID_CREDENTIAL`、`TARGET_NOT_FOUND`、`TARGET_UNAVAILABLE` 等错误语义不同，如果不在测试方案里明确区分，后续排障会把控制台登录问题和 API 调用问题混在一起。

## Risks / Trade-offs

- [真实联调需要稳定测试账号与可写测试数据] -> 本提案先假设联调环境至少能提供一套固定控制台账号、一个可浏览或管理的 API 资产，以及可回收的 API Key；若环境条件不足，apply 阶段需要先补齐环境约定。
- [当前自动化测试基础较薄，首轮补测工作量会比较大] -> 通过先做共享夹具和分层矩阵，避免每个模块各自散写测试。
- [部分闭环必须依赖真实后端数据落库和最终一致性] -> 这些步骤会保留人工冒烟与执行记录要求，不强行伪装成完全自动化。
- [顶层文档当前未必覆盖所有测试边界描述] -> 如果 apply 阶段发现现有 API 契约、统一前端规范或 `aether-console/DESIGN.md` 无法支撑测试实现，必须先同步更新顶层文档。

## Migration Plan

1. 先重新核对统一前端规范、`aether-console/DESIGN.md`、相关 API 契约和当前代码实现，确认本次测试范围不需要先改动顶层文档。
2. 建立测试矩阵、共享夹具和真实联调环境前置约定，优先打通自动化回归主路径。
3. 依次补齐认证、市场/工作台、API Key、Unified Access、调用日志等业务切片的自动化回归。
4. 在真实后端环境执行认证与核心闭环冒烟，记录通过项、失败项、环境依赖与清理结果。
5. 运行统一工程校验并沉淀一份可复用的联调测试记录，作为后续回归基线。

## Open Questions

- 当前提案先假设联调环境可以提供固定控制台账号、至少一个可用 API 资产以及可生成调用日志的可测上游；如果实施时这些条件不成立，需要在 apply 阶段补充环境准备任务。
- 当前提案先假设 API 调用日志可在一次 Unified Access 调用后于可接受时延内被查询到；如果后端存在异步延迟窗口，测试步骤需要在实现阶段补充等待与重试策略说明。
