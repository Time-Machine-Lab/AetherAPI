## Context

`aether-console` 当前已经实现 API 市场浏览、当前用户资产管理、API Key 管理、Unified Access Playground 和调用日志等 MVP 页面。现有实现符合项目要求的 Vue 3 + TypeScript + Vite 技术栈，也使用了 API 层、Composable、Pinia、i18n、shadcn-vue 和 `aether-console/DESIGN.md` 中的控制台语义角色规则。

本次问题主要不是“缺页面”，而是核心工作流不够顺：资产新建和编辑嵌在长页面中，用户容易失去上下文；AI 能力配置更新可能覆盖资产基础配置；市场卡片缺少可扫读元信息；用户不能直观看到平台统一调用地址；调用日志页面没有清晰区分“接口已提供的数据”和“当前契约尚未暴露的数据”。

后端架构文档明确当前产品是 API 市场与开发者控制台的一体化体验，不是独立管理后台。现有接口契约支持资产管理、市场发现、统一调用和调用日志摘要/详情，但 `api-call-log.yaml` 当前没有提供上游地址、请求体、响应体、请求 Header、响应 Header 等字段。

## Goals / Non-Goals

**Goals:**

- 将资产工作台优化为列表 + 新建模态框 + 编辑抽屉/模态框的聚焦式工作流。
- 绑定 AI 能力配置时保护用户已填写或已加载的资产基础配置。
- 为请求模板、请求示例、响应示例、上游地址、鉴权方式、AI 能力字段补充清晰说明。
- 在用户自然需要调用 API 的位置展示并支持复制 `/api/v1/access/{apiCode}` 平台调用地址。
- 使用语义化 Badge、Tag、图标和元信息组件增强市场卡片、资产列表和调用日志列表。
- 提供可复用的 JSON/代码展示组件，用于示例、Header、Payload、调用结果和结构化日志字段。
- 在调用日志详情中展示现有契约字段，并明确说明缺失诊断字段需要后续更新 API 契约。

**Non-Goals:**

- 不修复后端资产删除持久化、HTTPS 502、流式请求超时。
- 不实现真实 API 订阅功能。
- 在 `../docs/api/api-call-log.yaml` 更新前，不展示或伪造真实上游地址、请求体、响应体、请求 Header、响应 Header。
- 不引入新的 UI 框架，不绕开现有 API 层、Composable、i18n 和布局体系。

## Decisions

### 1. 资产编辑采用抽屉/模态框

资产列表应保持稳定，创建草稿和编辑资产通过聚焦式浮层完成。编辑已有资产优先使用抽屉，因为抽屉可以保留列表上下文，并承载较长的多分组表单；创建草稿可以使用更轻量的模态框。

备选方案是保留当前页内下方编辑区。它实现简单，但会造成页面跳动，用户容易丢失列表上下文，也让编辑状态和被编辑资产之间的关联不够明确。

### 2. 资产基础配置与 AI 能力配置分离管理

资产基础配置和 AI 能力配置应使用分离的本地表单状态。AI 能力绑定接口返回后，前端只合并本次需要更新的字段，不因为响应缺少某些字段而清空基础配置。

备选方案是每次接口返回后都用最新资产响应完整覆盖当前资产和所有表单。这种做法简单，但已经暴露出配置被清空的问题。

### 3. 抽取可复用展示组件

新增或整理小型共享展示组件：元信息行、状态/类型/方法标签、可复制地址、JSON/代码展示块。组件应继续使用 Tailwind、shadcn-vue 风格、lucide 图标、i18n 文案和既有设计 Token。

备选方案是在各页面临时堆 Tailwind class。短期更快，但会继续扩大当前视觉不一致的问题。

### 4. 平台调用地址由前端基于契约派生展示

统一调用地址可以基于当前 API base URL 和已记录的 `/v1/access/{apiCode}` 路径派生展示。UI 必须明确标注这是 AetherAPI 平台统一调用地址，而不是资产的上游地址。

备选方案是要求后端在每个资产详情中返回调用 URL。长期可以考虑，但当前契约已经足够支撑第一版展示。

### 5. 日志缺失诊断字段必须显式说明

调用日志页面应展示 `api-call-log.yaml` 当前提供的所有字段。对于用户期望但当前契约未提供的上游地址、请求体、响应体、Header 等字段，页面应显示“当前不可用/契约未暴露”的非错误态说明，而不是空白或伪造数据。

备选方案是完全隐藏这些区域。这样页面更简洁，但不能解释用户为什么找不到排查所需信息。

## Risks / Trade-offs

- [Risk] 资产编辑抽屉字段过多，信息密度过高。→ Mitigation：按基础信息、上游与鉴权、示例、AI 能力、生命周期操作分组。
- [Risk] 用户混淆平台调用地址与上游地址。→ Mitigation：明确使用“平台统一调用地址”和“上游地址”两个不同标签。
- [Risk] JSON 格式化遇到非法 JSON。→ Mitigation：非法 JSON 以纯文本展示，并给出非阻塞提示。
- [Risk] 卡片和列表元信息过多导致视觉噪音。→ Mitigation：使用紧凑图标元信息与语义标签，遵守控制台语义角色规则。
- [Risk] 用户仍然期待调用日志展示请求/响应体。→ Mitigation：前端显示契约边界说明，并把真实诊断字段作为单独的跨端变更处理。

## Migration Plan

1. 先补齐共享展示组件和必要测试。
2. 将资产工作台重构为列表 + 创建模态框 + 编辑抽屉/模态框。
3. 增加字段说明和 i18n 文案。
4. 在市场详情、资产详情和 Playground 路径中展示平台调用地址。
5. 优化市场卡片、资产列表和调用日志列表展示。
6. 使用统一 JSON/代码展示组件替换零散 `<pre>`。
7. 本变更为纯前端变更，无数据库迁移；如需回滚，可按页面/组件粒度回退。
