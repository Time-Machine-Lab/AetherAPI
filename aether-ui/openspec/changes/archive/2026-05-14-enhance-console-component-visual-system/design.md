## Context

`aether-console` 已经有基础 UI 组件、Tailwind 主题变量、shadcn-vue 组件目录和 `DESIGN.md` 中的控制台语义角色规则。但当前页面仍大量直接使用普通文字、基础 Card、Button、Input、Badge 和 `<pre>`，缺少“信息展示型组件”的统一规范。

这带来三个问题：

- 可扫读性弱：发布者、时间、方法、状态、分类等信息缺少图标、标签和分组。
- 语义混淆：只读状态有时看起来像按钮，操作按钮和状态标签边界不清楚。
- 复用不足：JSON 展示、字段说明、列表行、空态/错误态在不同页面各自拼样式。

本变更专门处理视觉与组件表达系统，不改变 API 契约和业务流程。由于它会形成新的页面视觉规则，必须先更新 `aether-console/DESIGN.md`，再进入具体组件和页面实现。

## Goals / Non-Goals

**Goals:**

- 补充 `aether-console/DESIGN.md` 中的控制台组件视觉表达规范。
- 建立元信息、状态标签、字段标题、字段组、列表行、JSON/代码展示、空态/错误态/不可用态的可复用组件模式。
- 明确只读状态、可点击操作、系统反馈、字段输入、数据展示之间的视觉区别。
- 让 API 市场、资产管理、调用日志、Playground 使用统一组件表达提升页面层次。
- 所有用户可见文案继续接入 i18n，图标优先使用 `lucide-vue-next`。

**Non-Goals:**

- 不改变任何后端接口契约。
- 不改变资产生命周期、订阅、鉴权、调用日志字段等业务能力。
- 不引入新的大型 UI 框架或设计系统依赖。
- 不重做整套品牌视觉，不偏离现有 Airbnb-inspired 控制台设计方向。

## Decisions

### 1. 先更新 DESIGN.md 再实现组件

由于本变更会新增跨页面视觉规则，必须先把组件角色、适用场景和禁用规则写入 `aether-console/DESIGN.md`。实现阶段以 DESIGN.md 作为权威依据。

备选方案是直接写组件。它速度更快，但会让视觉决策只存在于代码里，后续页面仍可能继续分裂。

### 2. 采用“展示型组件层”而不是扩展基础 Button/Card

新增组件应围绕展示语义设计，例如 `MetaItem`、`StatusTag`、`MethodBadge`、`FieldGroupHeader`、`CodeBlock`、`DataRow`、`EmptyState`。基础 Button/Card/Input 继续保留为底层原子组件，不承担业务状态展示职责。

备选方案是不断给 Button/Card 增加变体。这样容易让基础组件过重，也会继续造成只读状态像按钮的问题。

### 3. 所有元信息优先使用图标 + 文本组合

发布者、发布时间、更新时间、分类、请求方法、认证方式、耗时、状态码等短元信息应优先使用 lucide 图标配合文本或标签展示。图标必须服务信息识别，不作为纯装饰堆叠。

备选方案是继续纯文本展示。纯文本最简单，但不利于快速扫读，也难以形成层次。

### 4. JSON/代码展示独立成组件

请求示例、响应示例、Header、Payload、调用结果和结构化日志字段都应使用统一代码展示组件，支持格式化、复制、纯文本兜底、只读/可编辑区分和必要的折叠。

备选方案是各页面继续写 `<pre>`。这样会保持现有重复和粗糙感。

### 5. 页面改造按“组件替换优先”推进

具体页面不应大改业务逻辑，优先把原有文本和临时样式替换为新展示组件：市场卡片、资产列表、日志列表、Playground 结果区和表单字段组是第一批落地点。

备选方案是同时重构页面交互。交互重构已经由 `improve-console-frontend-ux-feedback` 覆盖，本变更只解决视觉表达系统。

## Risks / Trade-offs

- [Risk] 组件过多导致抽象膨胀。→ Mitigation：只抽取跨两个以上页面复用的展示模式。
- [Risk] 图标使用过度造成视觉噪音。→ Mitigation：图标只用于高频元信息和操作识别，禁止无意义装饰。
- [Risk] 与既有 `DESIGN.md` 中 Airbnb 风格冲突。→ Mitigation：继续使用白底、温暖阴影、克制主色和控制台语义角色规则。
- [Risk] 页面局部改造不一致。→ Mitigation：先落 DESIGN.md，再建立组件目录和示例用法，最后按页面逐步替换。

## Migration Plan

1. 更新 `aether-console/DESIGN.md`，新增组件视觉表达章节。
2. 新增展示型组件及必要工具函数。
3. 为组件补充 i18n 文案、交互状态和测试。
4. 分批替换 API 市场、资产工作台、调用日志、Playground 中的单一表达。
5. 运行 lint、type-check、test、build，并进行桌面/移动视口人工检查。
