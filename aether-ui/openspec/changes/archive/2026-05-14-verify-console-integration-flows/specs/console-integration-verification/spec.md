## ADDED Requirements

### Requirement: 控制台 MUST 为真实联调维护覆盖核心业务流的验证矩阵

`aether-console` MUST 为已经完成前后端对接的核心能力维护明确的联调验证矩阵，至少覆盖控制台会话认证、API 市场浏览、分类管理、资产管理、API Key 生命周期、Unified Access 调用以及 API 调用日志追踪，不允许只验证单一页面的静态渲染或局部接口连通性。

#### Scenario: 执行一次完整的控制台联调回归

- **WHEN** 团队需要在前后端对接完成后执行 `aether-console` 的联调回归
- **THEN** 验证范围 MUST 明确列出上述核心链路的成功路径与关键失败路径
- **THEN** 每条链路 MUST 能回溯到现有权威接口契约和实际页面入口
- **THEN** 每条链路的结论 MUST 被标记为通过、失败或受环境阻塞，而不是只记录“已测试”

### Requirement: 控制台 MUST 区分契约自动化回归与真实后端冒烟验证

`aether-console` MUST 采用分层验证策略：先执行契约对齐的自动化回归，再对依赖真实后端状态写入、跨接口串联和最终业务闭环的链路执行真实环境冒烟验证。

#### Scenario: 代码变更后执行控制台联调测试

- **WHEN** 团队为控制台联调能力补充或修改测试
- **THEN** 自动化回归 MUST 优先覆盖路由守卫、认证恢复、请求错误归一化、关键模块数据映射与页面状态切换
- **THEN** 真实后端冒烟 MUST 只承担本地 mock 或桩响应无法证明的跨系统行为验证
- **THEN** 在自动化回归未通过前，真实环境冒烟结果 MUST NOT 被视为最终验收结论

### Requirement: 控制台 MUST 在真实联调时显式确认环境边界

`aether-console` MUST 在真实后端联调执行时显式确认接口基地址、mock 开关、测试账号、前置测试数据与清理策略，避免把命中本地 mock 数据的结果误判为真实联调通过。

#### Scenario: 在真实后端环境执行一轮联调验证

- **WHEN** 团队准备对控制台执行真实后端冒烟验证
- **THEN** 测试记录 MUST 说明当前接口基地址、mock 开关状态、测试账号与前置数据准备情况
- **THEN** 若当前运行仍命中本地 mock 路径，则本轮验证 MUST 被判定为无效
- **THEN** 任何需要预置或清理的数据 MUST 记录准备方式与清理方式

### Requirement: 控制台 MUST 为联调失败保留可定位证据并区分错误边界

`aether-console` MUST 为联调失败保留可复现、可定位的证据，并明确区分控制台会话失效、Unified Access API Key 失败、业务数据为空、契约映射错误等不同问题边界。

#### Scenario: 核心业务链路在联调中失败

- **WHEN** 登录、工作台管理、API Key、Unified Access 或调用日志链路出现失败
- **THEN** 记录 MUST 至少包含失败步骤、入口页面、环境信息、稳定错误码或 `traceId` 与复现条件
- **THEN** 控制台会话失效 MUST 与 Unified Access API Key 失败分开记录与判断
- **THEN** 任何仍需人工复测或受环境阻塞的缺口 MUST 被显式列入未完成项，而不是隐藏在笼统备注中
