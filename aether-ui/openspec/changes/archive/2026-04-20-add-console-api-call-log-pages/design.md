## Context

当前仓库已经具备后端 Observability 一期的两个关键基础：

- `2026-04-19-observability-call-log-foundation` 已经建立调用日志事实写入与 `docs/sql/api_call_log.sql` 权威结构；
- `2026-04-19-observability-log-query-api` 已经建立当前用户日志列表与详情查询契约，并在根目录提供 `docs/api/api-call-log.yaml`。

但前端侧的 `aether-console` 还没有承接这条链路。现状上：

- `console-shell.ts` 的运营区只有 `credentials / usage / orders / billing / docs` 等导航建模，尚无 API 调用日志入口；
- `workspace.vue` 当前只针对 `#credentials` 做了独立分区渲染，其余 hash 仍落回分类与资产管理工作区；
- 现有 API 模块只覆盖 `catalog / credential / system / unified-access`，没有 observability 或 call-log 相关前端封装；
- 已归档前端提案明确把 observability / API 调用日志页面留作后续独立交付范围。

因此，这次设计的重点不是“发明一个全新的可观测性产品”，而是在现有控制台壳层、前端技术栈和视觉语义之内，为当前用户 API 调用日志补上一条最小、可联调、可验证的前端闭环。

约束来源如下：

- 接口与字段语义必须严格以 `../docs/api/api-call-log.yaml` 为唯一权威来源；
- 视觉与页面层级必须遵守 `aether-console/DESIGN.md` 中的控制台角色语义、Notice/Surface/Field 规则与列表/详情布局原则；
- 实现分层必须遵守 `../docs/spec/AetherAPI 前端技术栈与开发规范文档.md`，避免页面层直接发裸请求、避免新增不必要的全局 Store；
- 本次设计不新增或修改 `../docs/api/`、统一前端规范文档或 `aether-console/DESIGN.md`；如果实现阶段发现现有权威文档无法支撑新增交互模式，必须先补齐顶层文档。

## Goals / Non-Goals

**Goals:**

- 为 `aether-console` 增加当前用户 API 调用日志入口，并在控制台内完成列表、筛选、分页与详情查看闭环。
- 让前端展示严格对齐 `docs/api/api-call-log.yaml`，包括 `targetApiCode`、调用时间、耗时、结果分类、错误摘要与 AI 扩展字段。
- 复用现有受保护工作区、hash 导航、卡片语义、状态反馈与国际化结构，保持与现有 `credentials`、`catalog`、`playground` 一致的壳层体验。
- 让日志详情能够支持开发者排查，而不泄露其他用户数据范围，也不对不存在/越权场景给出超出契约边界的提示。

**Non-Goals:**

- 不引入管理员视角日志后台、全局监控面板、趋势图、聚合统计、导出能力或告警能力。
- 不展示完整原始请求/响应报文，不尝试在前端重建后端未暴露的调用上下文。
- 不顺带补齐 `usage`、`orders`、`billing` 或其他仍处于占位状态的控制台能力。
- 不修改后端 API 契约，不要求新增 `docs/api/` 或 `aether-console/DESIGN.md` 文档作为本次提案前提。

## Decisions

### 1. 日志能力落在现有受保护工作区中，以新的 hash 分区承载

决策：API 调用日志作为 `aether-console` 受保护工作区的一部分接入，优先采用与 `#credentials` 一致的 hash 分区模式，例如新增 `#api-call-logs` 导航入口，而不是新开第二个控制台应用或独立路由树。

原因：

- 现有控制台已经通过 `console-workspace` + hash 承接保护区运营能力，最小改动路径是沿用这一组织方式；
- `ConsoleLayout` 的激活态、侧边栏与页头逻辑已经天然支持 `routeName + hash` 模式；
- 这样可以避免为了一期日志浏览能力额外引入新的壳层、认证入口、布局副本和导航分叉。

备选方案：

- 新增独立页面路由，如 `console-call-logs`；
- 将日志页塞进现有 `usage` 占位项。

不采用原因：

- 独立路由会增加额外的页面结构与激活态判断成本，与当前最小交付目标不匹配；
- 复用 `usage` 会混淆“调用日志”和“用量统计”两个不同概念，不利于后续能力演进。

### 2. 前端数据访问继续遵守 `src/api + feature/composable + page` 分层

决策：新增独立的日志 API 模块与类型定义，页面层只负责渲染和交互编排；列表筛选、当前选中日志、详情加载、错误态与分页切换通过页面级组合逻辑或 feature 模块管理，不新增全局 Store。

原因：

- 现有 `catalog`、`credential`、`unified-access` 均已采用 API 模块封装请求，符合团队前端分层约束；
- API 调用日志是当前工作区内的局部业务状态，不适合提升为跨页面共享的全局状态；
- 这样便于后续为日志页补测试，也避免把请求参数转换和异常处理散落到页面模板中。

备选方案：

- 直接在 `workspace.vue` 里发请求并维护全部状态；
- 新增 Pinia store 管理日志列表与详情。

不采用原因：

- 页面层直接编排网络请求会突破既有分层规则；
- 全局 Store 会把只在日志工作区生效的局部状态过度中心化。

### 3. 页面结构采用“筛选条 + 列表 + 详情”的最小排查模型

决策：桌面端采用日志列表与详情并列的双区结构，顶部提供最小筛选条；移动端采用上下堆叠或列表进入详情的收敛式布局，但仍保持同一能力集合，不拆成多套业务流程。

筛选范围仅覆盖当前契约中已经存在的：

- `targetApiCode`
- `invocationStartAt`
- `invocationEndAt`
- `page`
- `size`

原因：

- `docs/api/api-call-log.yaml` 已经把一期查询闭环限定为列表、详情、目标 API 筛选和时间范围筛选；
- `aether-console/DESIGN.md` 已经提供页面搜索优先级、列表/详情区域对齐和状态反馈规则，可以直接复用；
- 这套结构既适合浏览最近调用，也适合点选单条查看错误摘要和 AI 扩展字段。

备选方案：

- 只做纯列表，不做详情面板；
- 做成多步骤日志检索向导或复杂分析仪表盘。

不采用原因：

- 纯列表不足以承接详情字段与排查需求；
- 向导式或分析型界面明显超出一期范围，也与当前契约不匹配。

### 4. 结果与错误语义以后端返回为唯一事实来源

决策：前端只负责把后端已经提供的 `resultType`、`success`、`httpStatusCode`、`error`、`aiExtension` 等字段做结构化展示，不推导额外业务结论，也不拼装未返回的调用语义。

具体包括：

- 列表页以摘要字段为准；
- 详情页在字段存在时展示错误码、错误类型、错误摘要、AI 供应商、模型、是否流式与用量快照；
- 404、越权或无权限范围场景统一按“当前用户无法查看该日志”处理，不提示是否属于其他用户。

原因：

- 后端提案明确“当前用户范围内日志”是唯一暴露视角，Consumer 仍为隐式内部概念；
- 写侧基础提案也明确一期不保存完整原始报文，因此前端不能假设可展示原始 request/response 内容；
- 保持只展示权威字段，可以减少前后端联调偏差与越权信息泄露风险。

备选方案：

- 在前端推导“失败原因分类”或“疑似上游异常”等二次标签；
- 用不同错误文案区分“不存在”和“越权”。

不采用原因：

- 会引入未被契约保证的解释层；
- 会增加数据范围泄露风险。

### 5. 与写侧基础能力的关系体现在可见字段与空态边界，而非前端直接对接写侧

决策：前端提案只消费查询契约，不直接面向 `observability-call-log-foundation` 设计任何写侧交互；写侧基础对前端的影响体现在“哪些事实字段会出现在日志页”和“何时可能出现空列表”。

原因：

- `observability-call-log-foundation` 明确不向前端直接暴露接口；
- 前端真正需要对接的是基于这份基础事实形成的查询 API；
- 这样可以把“两个后端提案对应的前端对接”收敛为一条实际可见的日志浏览链路，而不是错误地为写侧单独设计前端入口。

备选方案：

- 为写侧基础单独补“日志已记录”类前端能力；
- 在前端提案中额外定义日志创建/补录相关界面。

不采用原因：

- 与后端权威提案边界不一致；
- 也不符合当前一期日志只读查询能力的产品目标。

## Risks / Trade-offs

- [Risk] `console-workspace` 继续承载更多分区后，单文件条件分支可能变得拥挤。 → Mitigation：实现阶段把日志工作区抽成独立 feature/page section，`workspace.vue` 只保留 hash 到分区组件的装配逻辑。
- [Risk] 日志详情字段比列表显著更多，若直接全部平铺会破坏控制台可读性。 → Mitigation：优先展示核心摘要字段，把错误对象和 AI 扩展收敛到清晰的详情分组中，严格遵守 Surface 与 Status 语义。
- [Risk] 不区分“不存在”和“越权”可能让一部分排查体验不够“直接”。 → Mitigation：维持统一的边界错误提示，并在可能时显示后端返回的 `traceId` 或通用错误码，优先保证数据范围安全。
- [Risk] 后续若团队希望把 API 调用日志演进为更完整的 observability 中心，当前 hash 分区模式可能需要重构。 → Mitigation：在一期先最小闭环；若后续范围扩展为趋势、报表、告警，再通过独立 change 升级为单独导航与页面体系。

## Migration Plan

1. 在实现开始前，确认 `docs/api/api-call-log.yaml`、`aether-console/DESIGN.md` 与前端统一规范仍是当前权威来源。
2. 在 `aether-console` 中补齐日志查询 API 模块、DTO/类型映射与页面级编排逻辑，保证页面层不出现裸请求。
3. 为控制台增加 API 调用日志导航入口与工作区分区，并补列表、筛选、详情、空态、错误态与移动端收敛展示。
4. 补齐对应 i18n 文案与基础测试，验证分页、筛选、详情和错误反馈均严格对齐权威契约。
5. 若实现期发现需要新增视觉模式或补充顶层接口说明，暂停代码实现并先更新对应权威文档。

## Open Questions

- API 调用日志入口在侧边栏中应放在 `operations` 组内单独新增项，还是以更贴近 observability 语义的单独分组呈现；当前建议优先放在 `operations` 组，保持信息架构最小变更。
- 移动端日志详情是采用“列表下方展开”还是“选中后替换列表主体”的方式更符合现有控制台节奏；当前建议实现阶段优先选择更简单的上下堆叠方案。
- 是否需要在日志列表中同步提供 `resultType` 的颜色语义映射表；当前建议仅基于成功/失败与字段值做最小状态标识，不额外发明后端未定义的业务分级。
