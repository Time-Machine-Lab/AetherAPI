## Context

`aether-console` 当前已经具备 API 市场页、分类管理页和 API 资产管理页的基础信息架构，但页面实现仍以通用组件直接拼装为主，缺少控制台场景下的组件语义和版式层级。现状里最明显的问题有四类：

- 状态 badge、outline/ghost 按钮、公告条和胶囊型信息块共享近似的边框、阴影和圆角语言，导致信息状态与可执行操作难以区分。
- 顶部 shell 搜索、页面主搜索、资产卡片和详情卡片使用相近的提升层级，却没有清晰的主次关系，放大了“局部凸起”和页面重心漂移。
- 工作台中分类管理和资产管理的输入框、按钮、列表项与行内编辑态没有统一的高度和对齐基线，重命名态尤其显得压缩和突兀。
- 顶部公告区域只是复用普通 pill 容器，不具备系统反馈组件应有的强调方式和信息密度。

同时，本变更必须遵守几个约束：

- 不改动 `docs/api/*.yaml` 与既有业务流程，只刷新视觉层级、交互反馈和页面编排。
- 不绕开现有 `ConsoleLayout`、`src/components/ui/*`、`i18n` 和页面分层体系。
- 由于当前 `aether-console/DESIGN.md` 只有高层视觉语言，没有定义控制台场景的组件语义，因此必须先同步该文档，再进入代码实现。

## Goals / Non-Goals

**Goals:**

- 为 `aether-console` 建立一套明确的组件语义分层，区分状态、操作、公告、输入和容器的视觉角色。
- 修复 API 市场页和工作台页的布局权重失衡，让搜索区、卡片列表、详情面板和表单区拥有一致的对齐逻辑与空间节奏。
- 统一行内编辑、查询、创建和切换状态时的交互高度、边框、focus/hover 反馈与禁用态表现。
- 将新增的页面层级和组件语义沉淀到 `aether-console/DESIGN.md`，作为实现前置约束。

**Non-Goals:**

- 不新增后端接口、不调整 API 资产或分类管理的业务规则。
- 不切换到新的设计风格，只在现有 Airbnb-inspired 视觉方向内细化控制台语义。
- 不重做整套布局框架，不迁移到新的状态管理模式，也不改变登录、导航和路由边界。

## Decisions

### 1. 先更新 `aether-console/DESIGN.md`，再进行样式实现

- 决策：本次实现以前，先把控制台专属的组件语义、公告样式、搜索层级、表单对齐与状态反馈规则补充到 `aether-console/DESIGN.md`。
- 原因：当前设计文档只定义了整体氛围、色板、阴影和圆角，但没有规范“状态 badge 与按钮如何区分”“公告条如何强调”“行内编辑输入高度如何保持一致”等控制台特有问题；如果不先沉淀权威规则，后续页面改动会继续依赖局部判断。
- 考虑过的备选方案：直接在代码里扩展 `button` / `badge` / `input` 变体，再事后回填文档。
- 不采用原因：这会违反前端 OpenSpec 对顶层设计产物优先同步的要求，也不利于后续页面继续复用。

### 2. 以“语义角色”而不是“组件名字”来组织视觉区分

- 决策：围绕 action、status、notice、field、surface 五类语义角色整理视觉规则，而不是只给 `Button`、`Badge`、`Card` 各自零散加样式。
- 原因：当前混淆并不是单个组件写错，而是多个组件承担了相似外观却不同语义的职责。用语义角色建模，才能稳定回答“这个元素是在告知状态，还是允许点击，还是在发出系统通知”。
- 考虑过的备选方案：只微调 `Badge` 和 `Button` 的颜色或边框。
- 不采用原因：只修单个组件无法解决公告条、空状态、卡片容器和行内编辑之间的层级冲突。

### 3. 将 shell 级搜索降为辅助，把页面内搜索提升为内容主动作

- 决策：保留顶部搜索作为 shell 级快捷入口，但在视觉上降低其存在感；页面内搜索保留为当前页面的主操作，并与内容区容器对齐。
- 原因：两个搜索入口同时使用高提升圆角白底容器时，会造成页面主动作不明确。控制台首页与工作台实际依赖的是页面内搜索和管理表单，顶部搜索应服务于全局跳转/快速查找，而不是与页面主搜索竞争。
- 考虑过的备选方案：完全移除顶部搜索，或保持两个搜索入口同等强调。
- 不采用原因：移除顶部搜索会改变既有 shell 功能预期；维持同级强调则无法解决当前视觉重心冲突。

### 4. 用统一的对齐栅格处理列表、表单和行内编辑

- 决策：分类管理和 API 资产管理共享一套控制高度、按钮尺寸、行内操作间距和列表项内边距规则；重命名态不再临时压缩输入框，而是使用与标准字段同级的最小交互高度。
- 原因：当前工作台最突出的不协调感来自“单独为某个局部场景硬塞一个小尺寸输入框”。统一栅格能同时解决创建行、查询行、行内编辑和最近访问列表之间的错位问题。
- 考虑过的备选方案：仅单独把重命名输入框调高。
- 不采用原因：这只能修复一个局部症状，无法解决跨区域的按钮、输入框和卡片内部基线不一致。

### 5. 为公告条建立独立 banner 模式，而不是复用 pill

- 决策：顶部公告区使用独立 banner 容器，具备更明确的背景、边界、图标槽位和文案层级，并与 badge/tag/pill 在形态上拉开差异。
- 原因：公告承担的是系统反馈和当前重要信息提醒，不应与状态标识或导航胶囊共用同一视觉语法。
- 考虑过的备选方案：继续沿用当前圆角 pill，只调整颜色。
- 不采用原因：即使换色，也仍然会和 badge/tag、顶部 utility 项混淆。

## Risks / Trade-offs

- [组件变体数量上升] → 通过先更新 `aether-console/DESIGN.md` 和后续补充组件示例，控制变体增长为语义化扩展，而不是任意加皮肤。
- [更强的状态区分可能削弱当前“轻盈”观感] → 限制新增强调只用于 action/status/notice 三类关键角色，继续复用现有白底、暖灰文本和品牌红作为主基调。
- [顶部搜索降权后可能被误认为功能弱化] → 在 shell 中保留其位置与功能，但通过尺寸、阴影和背景层级与页面主搜索拉开主次，而非直接删除。
- [先改文档再改代码会拉长交付链路] → 这是受 OpenSpec 规则约束的必要顺序，也能减少后续实现返工。

## Migration Plan

- 第一步：更新 `aether-console/DESIGN.md`，明确控制台组件语义、公告条规范、搜索层级和对齐栅格。
- 第二步：基于文档更新 `Button`、`Badge`、`Input` 及相关容器样式的语义变体。
- 第三步：调整 `ConsoleLayout.vue`、`index.vue`、`workspace.vue` 的页面结构与样式落点，使其遵循新的层级和对齐规则。
- 回滚策略：如果刷新后的语义分层影响过大，可先保留 `DESIGN.md` 的规范结论，分批回退具体页面实现，而不回退权威文档中的问题定义与约束。
