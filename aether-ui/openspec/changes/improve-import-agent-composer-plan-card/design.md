## Context

Import Agent 工作区已经是单列对话式布局，流式过程中会展示 pending user turn、thinking 时间线、assistant reply 和当前计划卡片。当前输入框内容只在请求成功后清理；如果流式过程较长，用户会看到旧输入仍留在 composer 中。当前计划卡片则始终完整展示，尤其在计划较长、资产字段较多时，会挤压最新回复与思考过程的阅读空间。

前端现有实现把请求编排放在 `useImportAgentWorkspace`，页面展示放在 `ImportAgentWorkspace.vue`，文案走 `zh-CN` / `en-US` i18n。本次优化应继续遵守这个分层：输入清空和流式状态由 composable 维护，计划卡片展开 / 收缩由工作区组件消费状态并渲染。

## Goals / Non-Goals

**Goals:**

- 发送消息后立即清空 composer 文本，避免用户误以为消息尚未提交或重复发送。
- 让“当前计划”卡片支持用户手动展开 / 收缩。
- 流式回答开始时自动收缩计划卡片，回答完成并更新 session 后自动展开计划卡片。
- 保持计划内容和操作入口仍由最终 `session` 快照驱动。

**Non-Goals:**

- 不新增后端接口、请求参数或响应字段。
- 不把计划卡片状态持久化到后端。
- 不重做 Import Agent 工作区布局或拆分新页面。
- 不改变 thinking timeline、reply delta、session snapshot 的既有语义。

## Decisions

### 决策 1：发送时先缓存 payload，再立即清空输入框

`createSession` 和 `appendTurn` 必须先根据当前输入、附件和澄清答案构造请求 payload / pending summary，再清空对应文本字段。这样请求仍使用用户发送时的内容，UI 则立即进入“已发送”状态。

不采用“请求成功后再清空”的原因：流式请求期间输入框仍残留旧内容，会让用户对是否已提交产生困惑。

### 决策 2：计划卡片展开状态保存在工作区局部状态

计划卡片展开 / 收缩是纯前端展示状态，适合放在 `ImportAgentWorkspace.vue` 的局部 `ref` 或由 composable 暴露的 UI 状态中。考虑到它直接服务计划卡片展示与按钮交互，优先放在组件层，除非实现时发现需要被测试或跨组件复用，再下沉到 composable。

不采用后端持久化的原因：该状态不是业务事实，不应污染会话模型。

### 决策 3：自动收缩和自动展开以流式生命周期为触发点

当用户发送首条消息或追加轮次并进入流式状态时，计划卡片自动收缩；当流式完成并拿到最终 session 后，计划卡片自动展开。手动切换仍然可用，但下一次发送会再次按流式生命周期重置状态。

不采用“永远记住用户最后一次展开状态”的原因：用户在回答期间最需要聚焦最新流式内容，回答后最需要回到计划确认，两者优先级比历史偏好更明确。

### 决策 4：收缩态保留计划标题与关键状态

收缩后的计划卡片不应完全消失，应保留标题、版本、可执行 / 已确认状态和展开按钮，避免用户不知道计划区域去了哪里。展开态继续展示完整摘要、澄清控件、分类计划和资产计划。

不采用“模型回答时隐藏整个计划卡片”的原因：完全隐藏会造成布局跳动和上下文丢失。

## Risks / Trade-offs

- [发送后立即清空但请求失败，用户可能想找回原文] → pending turn 仍展示本次发送内容；失败提示保留，后续可视需要从 pending summary 或错误状态恢复。
- [自动展开 / 收缩与用户手动切换冲突] → 明确以“新一轮流式生命周期”为自动触发边界；流式期间和结束后用户仍可手动切换。
- [计划卡片收缩造成确认按钮暂时不可见] → 回答完成后自动展开，确认入口恢复；收缩态可保留状态标签提示。
- [布局在移动端跳动] → 使用现有卡片容器和轻量按钮切换内容区，避免改变主列宽度和整体布局结构。
