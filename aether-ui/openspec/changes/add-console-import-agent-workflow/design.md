## Context

当前 `aether-console` 已经具备与 import-agent 对接所需的大部分基础设施：受保护的 `ConsoleLayout`、基于 hash 的 `console-workspace` 分区入口、统一的 `axios` 实例、控制台登录态恢复、i18n 资源结构，以及按业务域拆分的 `src/api` 与 `src/features` 组织方式。与此同时，后端已在 `../docs/api/api-import-agent.yaml` 中定义当前用户导入会话创建、会话读取、轮次追加、计划确认、执行启动和执行详情查询接口，前端不需要发明新契约。

当前的主要约束有两个。第一，控制台壳层已经通过 `src/features/console/console-shell.ts`、`src/layouts/ConsoleLayout.vue` 和 `src/pages/workspace.vue` 建立了“单一受保护工作台 + hash 分区”的导航模型；如果另起一套独立路由，会破坏现有导航与壳层一致性。第二，后端当前没有“列出当前用户全部 import-agent 会话”的接口，只有按 `sessionId` 读取单个会话，因此前端如果想支持刷新后继续查看最近会话，必须自行保存最近一次活动会话标识，并在恢复失败时安全清理。

## Goals / Non-Goals

**Goals:**

- 在 `aether-console` 中提供与现有后端契约对齐的 import-agent 工作流，覆盖创建会话、追加澄清、查看计划、确认计划、启动执行和查看运行结果。
- 保持前端实现与现有控制台模式一致：继续复用 `console-workspace`、`ConsoleLayout`、`src/api` 分层、i18n 与 shadcn-vue 组件体系。
- 解决“无会话列表接口”前提下的前端会话恢复问题，使用户刷新页面或重新进入工作台时仍可回到最近一次活动会话。
- 为后续 apply 阶段提供清晰的模块落位、状态边界、错误处理和测试切片。

**Non-Goals:**

- 不新增或修改 `../docs/api/api-import-agent.yaml` 中的后端接口、字段或错误码。
- 不把 import-agent 扩展成跨应用能力；本次范围仅限 `aether-console`。
- 不绕开现有受保护工作台模型创建新的一级路由、独立布局或额外状态管理框架。
- 不在本提案中定义后端文件上传、服务端文档解析或多会话历史列表等后端未提供支撑的附加能力。

## Decisions

### 1. 将 import-agent 集成到现有 `console-workspace` 的新 hash 分区，而不是创建新的顶级页面

- 决策：新增一个受保护工作台分区入口，例如 `#import-agent`，并把侧边导航项放进现有 `Operations` 组，与 `credentials`、`api-call-logs` 等入口保持同层级。
- 原因：现有控制台已经把运营相关动作集中在 `console-workspace`，复用这一模型可以直接继承路由守卫、布局、标题与导航激活逻辑，减少额外页面编排和壳层分叉。
- 不选择“创建新的 `src/pages/import-agent.vue` 顶级路由”的原因：会额外引入路由命名、布局注入和导航状态同步成本，也与现有工作台以 hash 分区承载多业务流的模式不一致。

### 2. 采用“API 模块 + feature 工作区组件”的标准前端分层，而不是把请求和状态直接堆进页面

- 决策：在 `src/api/import-agent/` 下新增 `import-agent.api.ts`、`import-agent.dto.ts`、`import-agent.types.ts`，负责 DTO 映射和 HTTP 调用；在 `src/features/import-agent/` 下新增工作区组件与必要的 composable，承接页面状态、按钮行为与轮询逻辑；`src/pages/workspace.vue` 只负责按 hash 切换渲染。
- 原因：这与现有 `subscription`、`console-auth`、`api-call-log` 等业务域的组织方式一致，符合前端规范中“页面不直写裸请求、业务逻辑沉淀在 features/composables/api”要求。
- 不选择“直接在 workspace.vue 写完整流程”的原因：状态机会很快膨胀，难以测试，也不利于后续在不改动页面壳层的前提下继续扩展 import-agent 行为。

### 3. 以前端持久化最近活动 `sessionId` 的方式实现会话恢复，并在服务端读取失败时立即清理

- 决策：前端为当前登录用户保存最近一次活动的 import-agent `sessionId`，推荐以 `localStorage` 键按应用和用户维度隔离；当用户进入 `#import-agent` 分区或刷新页面时，如果存在缓存 `sessionId`，则调用 `GET /current-user/import-agent/sessions/{sessionId}` 恢复会话；若返回 404、401 或当前用户不可访问，则清理该缓存并回到“新建会话”初始态。
- 原因：后端目前没有会话列表接口，单靠页面内存状态无法支撑刷新恢复；而控制台现有认证与最近访问行为已经在使用本地持久化，这一策略与仓库现有模式兼容。
- 不选择“完全不做恢复”的原因：用户一旦刷新页面就会丢失当前会话上下文，体验与业务闭环都会受损。
- 不选择“只用 query 参数携带 sessionId”的原因：虽然 query 参数可分享当前上下文，但现有工作台主要依赖 hash 切分分区，额外引入 query 驱动会增加路由状态同步复杂度；实现阶段可把 query 作为增强项讨论，但不作为首轮必需能力。

### 4. 以服务端会话快照为单一真相源，前端只维护短暂交互状态

- 决策：创建会话、追加轮次、确认计划后，前端直接使用后端返回的最新 `ApiImportAgentSessionResp` 替换本地会话快照；启动执行后，以返回的 `ApiImportAgentRunResp` 作为当前 run 视图初值，并在执行中通过 `GET /current-user/import-agent/runs/{runId}` 轮询详情，同时在合适时机重新拉取会话详情同步 `status`、`confirmedPlanVersion` 与 `latestRunId`。
- 原因：后端已经把计划版本、确认门禁、会话状态和运行步骤结果建模为服务端状态，前端不应自行推导或缓存派生事实，否则容易在“追加澄清后确认失效”“运行启动后状态切换”这类场景中出现不一致。
- 不选择“前端自行拼装状态流转”的原因：这会把业务规则从后端复制一份到前端，后续维护成本和错配风险都更高。

### 5. 错误与交互反馈继续沿用控制台现有 i18n、统一错误归一化和卡片式布局语言

- 决策：import-agent 的用户可见文案全部进入 `src/locales/zh-CN/common.ts` 与 `src/locales/en-US/common.ts`；请求失败继续消费统一 `NormalizedHttpError`；布局和交互采用现有 `ConsoleLayout` 与 `aether-console/DESIGN.md` 的卡片、分栏和温暖型视觉语言，不单独定义新设计系统。
- 原因：前端统一规范明确要求用户可见文本接入 i18n，请求错误在 API/composable 层归一化，样式优先遵循目标应用 `DESIGN.md`。这条工作流是新业务切片，不是新品牌站点。
- 不选择“在组件中写硬编码提示或特例样式”的原因：会破坏现有控制台的一致性，也不利于后续文案迭代和多语言扩展。

### 6. import-agent 首屏改为“对话优先”消息输入，而不是独立建会话表单

- 决策：将原先按 `documentSource`、`documentSummary`、`importIntent`、`publisherDisplayName` 分散展示的建会话表单收敛为一个主消息输入区。用户的第一条消息直接创建会话；额外上下文折叠到“高级上下文”区域，仅在首次发送前可编辑。
- 原因：后端创建会话本就把 `importIntent` 视为首条用户消息，前端继续把这一动作包装成表单会打断会话心智；改成对话优先可以让“首次发送即建会话、后续继续澄清”的状态模型更清晰。
- 不选择“彻底删除可选上下文字段”的原因：现有后端契约仍然接受 `documentSource`、`documentSummary` 和可选发布者展示名，这些字段对导入质量仍有价值，应折叠而不是移除。

### 7. 本地文件附加采用浏览器侧文本提取，并映射到现有 create/append 契约

- 决策：前端在聊天输入区支持选择本地文本类文件，在浏览器侧读取文本后截取内容摘要。首次发送时把附加文件内容并入 `documentSummary` 与 `documentSource`；已有会话时，把文件摘要并入本次追加消息内容。整个过程不引入新的上传接口。
- 原因：当前后端没有文件上传或解析契约，但用户又需要把本地文档快速带入会话；浏览器侧文本提取可以在不改后端的情况下提供“附加文件”的真实体验。
- 约束：仅支持文本类文件、限制单文件大小与单次数量，并在 UI 中明确告知“不会直接上传原始文件”。
- 不选择“前端只保存文件名不读取内容”的原因：仅传文件名无法给 planner 足够上下文，用户仍需手动粘贴内容，达不到支持文件附加的目标。

### 8. 最新计划与运行结果内嵌到消息流，右侧只保留轻量会话摘要

- 决策：当前计划、确认动作、执行入口和最近一次运行结果不再占用独立的右侧大卡片，而是作为对话流中的结构化回复块显示在消息列表下方。右侧保留单一会话摘要卡，用于展示会话元数据、最新状态和快速刷新。
- 原因：import-agent 的核心心智已经从“表单驱动的多栏工作台”转为“与代理对话并逐步收敛计划”。把计划和运行放进消息流，可以让用户沿着一条主线理解“我说了什么 -> 代理给出什么计划 -> 我确认并执行 -> 得到什么结果”。
- 不选择“继续保留 plan/run 两张独立侧栏卡片”的原因：侧栏会让最新计划和最新运行与会话消息断开，用户需要来回切换主副区块，降低对话工作流的连贯性。

## Risks / Trade-offs

- [后端没有会话列表接口，前端只能恢复最近一次活动会话] -> 通过用户维度的本地 `sessionId` 缓存与失效即清理策略降低丢失上下文风险；如果后续需要历史会话列表，再由后端先补充权威契约。
- [运行结果依赖轮询而非推送] -> 轮询间隔和停止条件需要控制，避免无意义请求；执行阶段至少在 `EXECUTING` 时轮询，进入终态后立即停止。
- [单个工作台分区会承载创建表单、对话时间线、计划卡片和运行详情，复杂度较高] -> 通过 `src/features/import-agent/` 内部拆分子组件或 composable，保持 `workspace.vue` 和顶层工作区组件不失控。
- [本地缓存的会话可能已被后端删除或不再属于当前用户] -> 恢复失败时必须清理缓存并回到初始态，不能无限重试或展示陈旧信息。

## Migration Plan

1. 先确认 `../docs/api/api-import-agent.yaml`、前端统一规范和 `aether-console/DESIGN.md` 足以支撑本次实现；若发现缺口，先补权威文档。
2. 在 `src/api/import-agent/` 与 `src/features/import-agent/` 建立前端模块，并先补齐 DTO 映射、会话恢复和核心状态测试。
3. 将 import-agent 分区接入 `console-workspace` 与侧边导航，补齐 i18n、空态、错误态和确认门禁。
4. 完成执行启动与运行详情轮询，再补工作台级测试与工程校验。
5. 如果实现期间发现入口位置或恢复策略影响现有导航体验，可回退到“隐藏入口但保留 feature 模块”的中间状态，而不影响已有工作台分区。

## Open Questions

- 首轮实现是否需要把最近活动 `sessionId` 同步到 URL query，方便用户手动刷新后保留更强的可分享性？当前设计默认先只做本地恢复。
- 运行详情轮询是否需要在 UI 上暴露“手动刷新”作为补充入口，避免长轮询带来的等待感？当前设计倾向于两者并存，但以自动轮询为主。
