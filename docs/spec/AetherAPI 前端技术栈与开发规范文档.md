# AetherAPI 前端技术栈与开发规范

本文档适用于 AetherAPI 的官网、开发者门户、管理后台以及后续新增的 Web 前端应用。项目采用多仓模式，但所有前端仓库必须遵循统一的技术栈、工程结构、协作流程与 AI 代码生成约束。本文档同时服务于人类开发者与大模型协作编码场景，默认作为新增前端代码与重构工作的统一基线。

## 1. 核心原则 (Core Principles)

AetherAPI 前端开发遵循以下总原则：

- 一致性优先：跨应用保持统一的技术选型、目录结构、命名规则和代码风格，降低协作与维护成本。
- 复用优先：优先复用既有组件、设计 Token、请求封装、i18n 结构和业务模式，避免重复造轮子。
- 分层优先：页面、业务逻辑、全局状态、请求层和基础组件职责清晰，避免耦合蔓延。
- AI 可执行优先：规范必须足够明确，使大模型在生成代码时能稳定落在既定模式内，而不是自由发挥。

### 1.1 开发技术栈 (Tech Stack)

- **适用应用**：官网、开发者门户、管理后台，以及后续新增的 AetherAPI Web 前端应用。
- **工程模式**：采用多仓模式；每个前端应用可独立发布，但技术栈、目录结构与规范必须统一。
- **前端框架**：统一使用 `Vue 3 + TypeScript + Vite`，默认采用 Composition API 与 `<script setup lang="ts">`。
- **样式方案**：统一使用 `Tailwind CSS` 作为首选样式方案；主题能力通过 CSS Variables 与设计 Token 统一管理。
- **组件体系**：统一使用 `shadcn-vue` 作为基础 UI 组件体系；跨应用共享组件风格、交互模式和封装方式。
- **路由方案**：统一使用 `vue-router + unplugin-vue-router`，采用文件式路由；所有页面路由默认懒加载。
- **状态管理**：统一使用 `Pinia` 管理跨页面、跨模块共享的全局状态。
- **请求方案**：统一使用 `axios` 实例作为请求入口，统一处理鉴权注入、错误归一化、超时与重试策略。
- **国际化方案**：统一使用 `i18n` 方案管理多语言能力，默认支持 `zh-CN` 与 `en-US`。
- **工程提效**：统一使用 `autoimport` 提升开发效率，但自动导入范围必须受控。
- **共享基础能力**：所有前端应用默认共用 UI 组件体系、设计 Token、请求封装、Pinia 约定、i18n 结构以及鉴权与权限控制模式。

### 1.2 Git 与代码流 (Git Workflow)

- **仓库策略**：各前端应用独立维护仓库，但必须遵循统一的提交规范、质量门槛和发布规则。
- **分支策略**：默认以 `main` 作为稳定分支，开发工作从短生命周期分支发起，命名建议使用 `feature/*`、`fix/*`、`chore/*`。
- **提交规范**：统一采用 Conventional Commits。
- **允许的提交类型**：`feat:`、`fix:`、`refactor:`、`docs:`、`style:`、`test:`、`build:`、`chore:`。
- **提交示例**：`feat: add API marketplace search filters`、`fix: correct portal token refresh flow`。
- **合并门槛**：代码合并前至少通过 `lint`、`format --check`、`type-check`、`build`。
- **发布原则**：各仓库独立构建和部署，但提交说明、分支策略和发布前检查保持一致。

### 1.3 命名与代码风格 (Naming & Code Style)

- **代码风格基线**：统一使用 Composition API 与 `<script setup lang="ts">`；禁止在同一仓库中混用 Options API、Class 风格或多套不一致的代码组织方式。
- **目录结构**：各前端仓库默认采用以下结构：
  - `src/app`：应用启动、全局 Provider、路由和 i18n 装配。
  - `src/pages`：页面级路由组件，仅负责页面编排与数据拼装。
  - `src/layouts`：全局布局组件。
  - `src/components`：通用基础组件与共享 UI 组件。
  - `src/features`：按业务域组织的功能模块、业务组件与领域逻辑。
  - `src/stores`：Pinia 全局状态。
  - `src/api`：按业务域拆分的 API 模块、DTO 和接口类型。
  - `src/composables`：可复用组合式逻辑与请求编排。
  - `src/utils`：纯工具函数。
  - `src/locales`：国际化资源文件。
  - `src/assets`：静态资源。
- **职责边界**：
  - 页面组件负责页面装配、状态组合和交互编排，不直接发起裸请求。
  - 通用组件不承载业务请求逻辑。
  - 业务逻辑优先沉淀在 `features`、`composables`、`api` 中。
  - `Store` 仅承载全局共享状态，不直接耦合视图组件。
- **跨模块引用**：允许跨模块引用，但必须避免循环依赖；对于需要长期复用的能力，优先通过稳定导出入口暴露。
- **组件命名**：组件文件统一使用 `PascalCase.vue`，例如 `UserProfileCard.vue`。
- **Composable 命名**：统一使用 `useXxx.ts`，例如 `usePortalSession.ts`。
- **Store 命名**：统一使用 `useXxxStore.ts`，例如 `useAuthStore.ts`。
- **API 模块命名**：统一使用 `xxx.api.ts`，按业务域拆分，例如 `marketplace.api.ts`。
- **类型命名**：DTO 文件统一使用 `xxx.dto.ts`，领域类型文件统一使用 `xxx.types.ts`。
- **工具函数命名**：函数与变量统一使用 `camelCase`，常量统一使用 `UPPER_SNAKE_CASE`。
- **路由文件规范**：页面文件放置于 `src/pages`，遵循 `unplugin-vue-router` 约定；优先使用文件式路由，不手写分散的路由表。
- **路由元信息**：页面路由 `meta` 至少应包含 `titleKey`、`layout`、`requiresAuth`。
- **布局约定**：布局值统一限制为 `MarketingLayout`、`PortalLayout`、`AdminLayout`。
- **认证约定**：官网默认匿名访问；开发者门户和管理后台通过 `meta.requiresAuth` 接入统一鉴权逻辑。
- **i18n 规则**：
  - 所有用户可见文案必须接入国际化。
  - 严禁在组件模板、脚本或配置中硬编码中文或英文文案作为最终展示文本。
  - 国际化资源文件按领域拆分。
  - 资源语言统一使用 `zh-CN` 和 `en-US`。
  - 国际化 Key 使用领域化命名，例如 `portal.user.profile.title`。
- **请求规范**：
  - 所有请求必须通过统一的 `axios` 实例发起。
  - 请求模块按业务域拆分，禁止在页面组件中直接写裸请求。
  - 请求地址必须通过环境变量与统一配置注入。
  - 禁止在业务代码中硬编码生产域名、测试域名、完整 URL 或绝对站内跳转路径。
- **路径规范**：
  - 页面跳转优先使用命名路由或相对路径策略。
  - `@/` 这类模块导入别名允许使用，不在“禁止绝对路径”范围内。
- **Pinia 规范**：
  - 仅存放全局共享状态，页面局部状态留在组件内部。
  - Store 必须按业务领域拆分。
  - 异步请求与接口编排放在 `api` 或 `composables`，不直接堆在 Store 中。
- **Auto Import 规范**：
  - 允许自动导入 Vue、Vue Router、Pinia 和通用组合式 API。
  - 不允许将业务 API、业务组件或领域依赖配置为隐式自动导入。
- **样式规范**：
  - 样式优先使用 Tailwind utility class。
  - 复杂样式优先沉淀为组件、语义化 class 或设计 Token，不滥用零散的局部样式。
  - 视觉实现必须与 `shadcn-vue` 组件体系保持一致。
- **图标规范**：
  - 严禁在 Vue 模板内直接编写 SVG 图标。
  - 如果需要自定义 SVG，必须先封装为单独图标组件，再在业务组件中引用。

### 1.4 其他开发事项 (Other Conventions)

- **错误处理规范**：
  - 请求错误统一在 `axios` 实例、`api` 层或 `composables` 层归一化处理。
  - 页面层只消费标准化后的成功/失败状态，不直接解析底层错误对象。
  - 需要暴露给用户的错误提示也必须接入 i18n。
- **AI 协作约束**：
  - 新增代码默认优先复用现有目录结构、组件封装和业务模式。
  - 禁止混用多种实现风格；同类问题优先沿用仓库既有解法。
  - 请求必须走统一 API 层。
  - 用户可见字符串必须走 i18n。
  - 新页面、新组件、新 Store、新 API 模块必须遵循本文命名规则与落位规则。
  - 生成样式时优先使用 Tailwind utility；只有在复用或表达复杂样式时才抽离为组件或语义类。
  - 生成图标时必须优先引用已有图标组件，不直接内嵌 SVG。
- **共享能力演进原则**：
  - 对多个应用通用的能力，优先沉淀为共享规范或共享实现，而不是在各仓库重复实现。
  - 对仅在单一业务域生效的逻辑，保持在本应用、本领域内封装，避免过度抽象。

## 2. 治理与维护 (Governance & Maintenance)

### 2.1 测试规范 (Testing Strategy)

- **强制质量检查**：所有仓库必须通过 `ESLint`、`Prettier`、`TypeScript` 检查。
- **本地提交前检查**：建议通过 `husky + lint-staged` 在提交前执行最小质量校验，至少覆盖 `lint` 与格式检查。
- **合并前校验**：代码合并前至少完成 `lint`、`format --check`、`type-check`、`build`。
- **单元测试策略**：当前阶段单元测试为推荐项，优先覆盖以下内容：
  - 复杂 `composables`
  - 共享工具函数
  - 请求适配层
  - 复杂全局状态逻辑
- **测试工具建议**：推荐使用 `Vitest` 作为单元测试框架；如需组件测试，可结合 Vue 生态测试工具扩展。
- **补测原则**：涉及鉴权、路由守卫、计费逻辑、请求签名、核心流程编排等高风险变更时，应补充对应测试。

### 2.2 部署与 CI/CD (Deployment & CI/CD)

- **环境划分**：统一采用 `local`、`dev`、`test`、`prod` 四套环境。
- **环境变量规范**：
  - 前端公开配置统一通过 `Vite` 环境变量注入。
  - API 基地址、站点跳转配置、埋点配置等必须通过环境变量管理。
  - 禁止在业务代码中硬编码环境相关地址。
- **构建流程**：每个前端仓库独立构建，构建脚本、环境命名和质量门槛保持统一。
- **CI 最小流水线**：`install` -> `lint` -> `format --check` -> `type-check` -> `build`。
- **发布流程**：
  - 功能开发在特性分支完成。
  - 合并到稳定分支前完成代码检查与必要评审。
  - 合并后由各仓库流水线独立完成对应环境的构建与部署。
  - 生产环境发布建议通过版本标记或人工确认步骤执行，保证可追溯性。
- **多应用发布原则**：官网、开发者门户、管理后台可以独立发布，但共享的规范、配置命名和接入方式必须一致。

### 2.3 监控与告警 (Monitoring & Alerting)

- **前端监控范围**：至少覆盖运行时异常、接口请求失败、关键页面白屏、核心操作失败率。
- **日志字段规范**：前端异常日志与请求失败日志应尽量携带应用标识、路由信息、版本号、环境信息与请求链路标识。
- **敏感信息处理**：日志与告警中严禁输出令牌、密钥、隐私数据和完整敏感请求体。
- **错误上报原则**：同类错误应归并上报，避免重复噪音；高频错误应能快速定位到具体应用、页面和版本。
- **联调要求**：前后端联调时，接口错误必须尽量保留可追踪标识，便于跨系统排查。
