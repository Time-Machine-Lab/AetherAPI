# AetherAPI 第一期项目规划书

## 1. 阶段背景

### 1.1 规划背景

AetherAPI 的长期方向已经明确：打造一个以 AI 体验为差异化能力的 API Hub 与统一接入平台，逐步延伸到 API 市场、AI 模型聚合、MCP 生态与商业化能力。

但从第一期仅有 1 周开发周期的现实出发，我们不应该直接铺开“API 市场 + 积分体系 + AI 文档转换 + MCP Gateway + 多模型聚合 + 开发者门户”的完整愿景，而应该优先完成一个可以被真实使用、真实验证、真实复用的最小闭环。

本期规划的核心思路是：

- 第一期定位为 `AI-ready API Hub MVP`
- 产品策略坚持 `Hub-first, Gateway-light`
- 不追求“大而全”的平台形态
- 优先验证“API 聚合管理 + 统一接入 + 可被调用 + 可观测”这条主链路
- 为后续 AI 文档转换、MCP Gateway、积分与商业化能力打下稳定基础

### 1.2 本期周期

- 周期：1 周
- 目标类型：第一期基础能力建设
- 交付导向：可演示、可联调、可复用、可作为下一期迭代基座

---

## 2. 本期目标

### 2.1 阶段目标定义

第一期不以“完整商业化平台上线”为目标，而以“完成 AI-ready API Hub 最小闭环”为目标。

本期结束时，AetherAPI 应至少具备以下能力：

1. 平台可以录入并管理一批 API 资源
2. 使用方可以通过最小展示页查看 API 列表、查看 API 详情、获取调用凭证
3. 平台可以通过轻量统一接入层完成 API 调用转发，而不是重做底层网关
4. 平台可以记录基础调用日志，便于后续统计、计费和排障
5. 平台至少接入 2 到 3 个真实 API，其中至少 1 个为 AI API 样例
6. 团队可以基于这套骨架继续扩展 AI 文档解析、积分系统和 MCP 能力

### 2.2 本期成功标准

若本期完成顺利，至少应满足以下验收结果：

- 能成功接入 2 到 3 个真实 API 样例
- 其中至少有 1 个 AI API 样例完成接入验证
- 能通过统一入口完成一次完整调用
- 能查看调用结果、调用状态与基础日志
- 能由后台维护 API 元数据，而不是写死在代码中
- 能通过最小展示页完成 API 浏览与基础体验演示
- 能支撑内部团队将后续新 API 继续按同样方式接入

---

## 3. 范围界定

### 3.1 本期建议聚焦范围

结合项目概念文档中的 MVP 方向，以及“一周开发周期”的现实约束，建议第一期聚焦以下 6 个模块。

| 模块 | 本期目标 | 价值说明 |
|------|---------|---------|
| API 资源管理 | 完成 API 基础信息录入、分类、状态管理、详情展示 | 建立平台的最小内容底座 |
| Hub 最小展示页 | 提供 API 列表与详情展示页，支撑浏览与演示 | 体现 AetherAPI 作为 Hub 的产品形态 |
| 开发者接入与凭证管理 | 提供调用凭证生成与校验能力 | 建立统一鉴权入口 |
| 轻量统一接入层 | 提供统一调用入口，封装 API 转发与调用控制 | 验证“统一接入”核心价值，避免重做网关 |
| 调用日志与基础监控 | 记录请求、响应、耗时、状态、错误信息 | 为调试、统计、后续计费提供基础数据 |
| AI-ready 能力预留 | 支持 AI API 类型标记、流式能力标记与计费字段预留 | 保留产品的 AI 发展方向，不把一期做成普通 API 平台 |

### 3.2 本期暂不纳入范围

以下内容非常重要，但不建议在第一期一周周期内重度展开：

- 完整积分充值、消费、提现闭环
- AI 文档自动解析与代码自动生成
- MCP Gateway 对接 Claude / Cursor
- 多 AI 模型统一聚合与智能路由
- 完整 AI 网关能力，如模型路由、Fallback、统一 Token 计费
- 开放市场交易、订阅套餐、平台抽成规则
- 企业级权限体系、审计体系、多租户体系

### 3.3 取舍原则

本期取舍依据如下：

- 优先做“主链路基础设施”，不优先做“高级体验功能”
- 优先做“可运行能力”，不优先做“复杂运营系统”
- 优先做“Hub 形态验证”，不优先做“完整市场化闭环”
- 优先做“轻量统一接入”，不优先做“重网关研发”
- 优先保留“AI-ready 扩展能力”，不强求本期交付完整 AI 平台能力
- 优先为下一期留接口，不强求本期一次做完

## 4. 本期交付方案

### 4.1 模块一：API 资源管理

#### 目标

建立平台可维护的 API 注册中心，替代散落文档或硬编码配置。

#### 建议功能点

- API 基础信息管理
- API 分类管理
- API 状态管理（草稿、启用、停用）
- API 详情信息维护
- 上游服务地址与请求配置保存
- API 类型标记（普通 API / AI API）
- AI 能力字段预留（是否流式、供应商、模型或能力标签）

#### 建议数据字段

- API 名称
- API 唯一标识
- 分类
- 描述
- 请求方法
- 路径
- 上游地址
- 鉴权方式
- 请求示例
- 响应示例
- 状态
- API 类型
- 是否支持流式
- AI 供应商
- 模型名或能力标签
- 计费预留字段

#### 验收标准

- 后台可新增、编辑、停用 API
- 前台页面或接口可查询 API 列表和详情
- 至少完成 2 到 3 个 API 的录入与可见化

### 4.2 模块二：Hub 最小展示页

#### 目标

让第一期交付结果不仅可调用，也具备基础的 Hub 产品形态，便于演示、浏览和内部试用。

#### 建议功能点

- API 列表页
- API 详情页
- 基础分类或标签展示
- 调用方式或凭证获取说明
- 区分普通 API 与 AI API

#### 本期建议简化

第一期只做最小展示页，不做复杂运营后台、搜索排序优化和完整市场交易体验。

#### 验收标准

- 可以通过页面查看 API 列表和详情
- 页面上能区分普通 API 与 AI API
- 页面可用于项目演示和内部试用说明

### 4.3 模块三：开发者接入与凭证管理

#### 目标

让平台具备“谁可以调、拿什么调、如何识别调用方”的基础能力。

#### 建议功能点

- 调用方身份模型
- API Key 生成
- 凭证与调用方绑定
- 基础鉴权校验中间件
- 凭证启停控制

#### 本期建议简化

为了保证一周内可交付，第一期建议先采用 API Key 模式，不在本期引入复杂 OAuth、细粒度权限和多角色体系。

#### 验收标准

- 平台可为一个调用方生成凭证
- 调用统一入口时能校验凭证有效性
- 无效凭证调用会被正确拒绝

### 4.4 模块四：轻量统一接入层

#### 目标

验证 AetherAPI 最核心的产品价值：通过统一入口访问不同 API，同时保持 `Gateway-light` 的实现边界。

#### 建议功能点

- 统一调用入口
- 路由到目标 API 配置
- 请求参数透传
- 基础超时与错误处理
- 标准化返回结构
- 至少支持 1 个 AI API 的接入调用
- AI API 的流式能力预留或基础透传支持

#### 本期建议实现边界

第一期重点不是自研高性能网关，而是以“服务层统一代理”或“轻量封装成熟网关”的方式打通调用流程，先验证产品主链路。

#### 验收标准

- 调用方使用统一入口可以成功请求目标 API
- 支持 GET / POST 两种常见方式至少一种完整打通
- 至少完成 1 个 AI API 的统一接入验证
- 调用失败时平台能返回清晰错误信息

### 4.5 模块五：调用日志与基础监控

#### 目标

建立最基础的可观测性，支撑排障和后续运营分析。

#### 建议功能点

- 调用记录落库
- 记录调用方、目标 API、请求时间、耗时、状态码
- 错误信息记录
- 基础查询能力
- 为后续积分和 AI 计费预留字段

#### 本期建议简化

第一期先不做完整监控看板与告警系统，以“日志可查”为最低交付标准。

#### 验收标准

- 每次调用都能留下记录
- 可以按 API 或调用方查询日志
- 失败调用可看到基础错误原因

### 4.6 模块六：AI-ready 能力预留

#### 目标

确保第一期虽然不重做 AI 平台能力，但已经具备明确的 AI 演进基础，不会沦为普通 API 管理后台。

#### 建议功能点

- API 元数据支持 AI 类型区分
- 支持 AI API 样例接入
- 支持流式接口能力标记，必要时透传流式响应
- 在 API 配置和日志中预留计费相关字段
- 为后续 MCP 工具描述结构预留扩展点

#### 本期建议简化

第一期不落地完整 MCP Server、多模型路由、统一 Token 计费，只完成数据结构和样例验证。

#### 验收标准

- 至少 1 个 AI API 能完成从配置到调用的闭环验证
- API 管理中可以区分 AI API 与普通 API
- 文档或数据结构中明确保留流式、计费、MCP 描述扩展位

---

## 5. 本期不建议硬上的功能

### 5.1 AI 文档转换

这是 AetherAPI 的重要差异化能力，但第一期直接落地会牵涉文档解析、模型调用、格式统一、错误兜底等较复杂问题。如果一周周期内强行推进，很容易挤占主链路建设时间。

建议第一期只做一项铺垫工作：

- 预留 API 导入数据结构
- 支持手工录入或半结构化录入

第二期再推进：

- OpenAPI / Postman 导入
- AI 自动补全说明
- 调用代码自动生成

### 5.2 积分系统

积分系统与商业化强相关，但它的前提是平台已经具备稳定调用链路与可追踪日志。否则计费基础不可靠，后续反而会返工。

建议第一期只做以下预留：

- 在调用日志中保留可计费字段
- 在 API 配置中预留计费配置字段

### 5.3 MCP Gateway

MCP 是很好的后续亮点，但它依赖平台底层 API 发现、鉴权、调用封装都先稳定下来。第一期更适合先把 MCP 所依赖的“标准工具描述能力”和“统一调用层”做扎实。

### 5.4 完整 AI 网关能力

AI 网关是 AetherAPI 后续非常重要的方向，但第一期不建议同时引入多模型统一路由、Fallback、统一 Token 计费、复杂限流和可观测体系，否则会明显冲击交付节奏。

建议第一期只做：

- 至少 1 个 AI API 样例接入
- AI 类型元数据和流式能力预留
- AI 调用日志字段预留

## 6. 一周排期建议

### 6.1 排期拆分

| 时间 | 建议任务 | 预期结果 |
|------|---------|---------|
| 第 1 天 | 梳理领域模型与接口结构，完成数据库表设计 | 明确 API、调用方、凭证、日志四类核心模型 |
| 第 2 天 | 完成 API 资源管理能力与 AI-ready 字段设计 | 可新增、编辑、查询 API 元数据，并支持 AI API 标记 |
| 第 3 天 | 完成凭证生成与鉴权中间件 | 可为调用方生成 API Key 并完成校验 |
| 第 4 天 | 完成轻量统一接入层与基础转发 | 可通过统一入口调用目标 API |
| 第 5 天 | 接入 2 到 3 个真实 API，并完成 1 个 AI API 样例验证 | 验证普通 API 和 AI API 的统一接入能力 |
| 第 6 天 | 完成日志记录、最小展示页、错误处理与联调验证 | 可展示 API 列表与详情，并查询调用记录 |
| 第 7 天 | 补充文档、统一验收与演示材料整理 | 形成第一期交付成果与下一期输入 |

### 6.2 并行建议

如果团队投入允许，可以按以下方式并行：

- 一人负责 API 资源管理与最小展示页
- 一人负责统一接入链路与鉴权
- 一人负责日志、样例 API 接入和文档整理

如果人力有限，则建议严格按“数据模型 -> 资源管理 -> 鉴权 -> 统一接入 -> 样例 API -> 日志与展示”顺序推进。

## 8. 风险与应对

### 8.1 主要风险

| 风险 | 说明 | 应对方式 |
|------|------|---------|
| 范围失控 | 想一次性做完 AI 文档转换、积分、MCP 等多个亮点 | 坚持只做最小闭环，额外内容只做预留 |
| 上游 API 差异大 | 不同 API 的认证与参数结构不一致 | 第一期先选 2 到 3 个结构简单的样例 API，并只接入 1 个 AI API 样例 |
| 网关复杂度被低估 | 若一开始追求完整网关能力，周期会失控 | 先做轻量统一代理与配置驱动调用 |
| 前台页面范围扩张 | 最小展示页若演变成复杂前端，会拖慢主链路交付 | 只保留列表、详情、基础说明三类页面 |
| 计费设计过早介入 | 没有稳定日志前，商业规则容易返工 | 第一期只预留字段，不落地完整计费 |

### 8.2 依赖前提

本期顺利推进依赖以下前提：

- 明确当前技术栈与服务边界
- 提前选定 2 到 3 个演示用 API
- 提前选定 1 个 AI API 样例
- 明确最小展示页只服务于浏览、演示和内部试用

## 9. 下一期衔接建议

如果第一期按计划完成，第二期建议优先进入以下方向：

1. OpenAPI / Postman 导入能力
2. AI API 接入增强与流式能力完善
3. 基础计费字段到积分扣减闭环
4. 调用代码示例自动生成
5. MCP Gateway 最小接入版本
6. API 市场搜索、分类和详情体验增强

## 10. 讨论结论与建议

结合当前概念文档与本次讨论结果，我建议 AetherAPI 第一期的明确主题定义为：

**“完成 AetherAPI 的 AI-ready API Hub MVP，验证 API 聚合管理、统一鉴权、轻量统一接入、调用日志追踪，以及最小 AI API 接入能力。”**

如果我们认可这个主题，那么本期最值得优先完成的不是“平台大而全亮点”，而是下面这 6 件事：

1. API 资源管理
2. Hub 最小展示页
3. 开发者凭证管理
4. 轻量统一接入层
5. 调用日志与基础监控
6. 至少 1 个 AI API 样例接入

这套规划的好处是：

- 一周内真正可落地
- 能体现 AetherAPI 不是普通 API 管理后台，而是 AI-ready 的 API Hub
- 能直接支撑内部团队开始试用
- 为第二期 AI 化和商业化功能提供稳定底座
- 降低第一期因范围过大而交付失控的风险

---

## 11. 当前落地建议

基于当前已经确认的方向，第一期建议按以下共识推进：

1. 定位明确为 `AI-ready API Hub MVP`
2. 产品策略明确为 `Hub-first, Gateway-light`
3. 第一期开启最小展示页，而不是只做纯后台接口
4. 第一期接入 2 到 3 个真实 API，其中至少 1 个为 AI API 样例
5. 积分系统、MCP、AI 文档转换和完整 AI 网关能力只做预留，不做重交付

如果严格按以上边界执行，第一期最有机会产出一个可展示、可试用、可继续迭代的版本。
