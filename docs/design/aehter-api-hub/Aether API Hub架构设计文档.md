# Aether API Hub架构设计文档

## 1. 架构定位

Aether API Hub 第一阶段是一个面向开发者的 API 市场与统一调用中枢，不是平台内部运维录入系统，也不是重型网关产品。

本期明确坚持两个产品理念：

- `everyone can share API`
- `everyone can use API`

因此，Aether API Hub 的职责不是替用户代持 API，而是统一管理用户发布到市场中的 API 资产，并提供统一发现、统一鉴权、统一转发与调用日志能力。

第一期架构定位为：

- 一个 `AI-ready API Hub MVP`
- 一个 `Hub-first, Gateway-light` 的模块化单体
- 一个同时承载 API 市场与开发者控制台的后端系统

## 2. 系统上下文

### 2.1 对外角色

- 资产发布者：在平台中创建、维护并发布自己的 API 资产
- 资产使用者：在 API 市场中浏览并调用已发布的 API 资产
- 开发者控制台用户：查看自己的 API Key、调用记录、资产工作台等信息

第一期没有独立管理端。管理员和普通用户都在同一套 API 市场与开发者控制台中完成对应操作。

### 2.2 外部依赖

- 上游 API 服务
- AI Provider API
- MySQL
- TML SDK Result 统一返回能力

当前阶段尚未接入独立 API Gateway，统一调用链路由 Hub 内部的 Unified Access 模块直接完成最小可用转发。

## 3. 核心模块

### 3.1 API Catalog

负责 API 资产主数据与资产生命周期。

核心职责：

- 当前用户创建自己的 API 资产
- 当前用户维护自己的 API 资产配置
- 当前用户发布、下架、删除自己的 API 资产
- 为市场发现、统一调用、日志记录提供统一资产主数据

资产生命周期以 `DRAFT / PUBLISHED / UNPUBLISHED` 表达，不再使用平台运营语义的 `ENABLED / DISABLED`。

### 3.2 API Marketplace

负责市场浏览与统一入口体验。

核心职责：

- 展示已发布 API 资产列表与详情
- 区分普通 API 与 AI API
- 为用户提供统一调用地址与接入说明

市场只暴露已发布且未删除的资产，不暴露资产工作台内部字段、上游地址密钥等写模型信息。

### 3.3 Developer Console

负责当前登录用户的控制台能力。

核心职责：

- 我的 API 资产工作台
- API Key 管理
- API 调用日志查询
- 账号当前用户上下文下的业务操作

开发者控制台与市场是同一产品的一体两面，不拆分管理端。

### 3.4 Consumer & Auth

负责两类鉴权链路：

- 控制台用户登录态鉴权：用于控制台业务接口
- API Consumer 鉴权：用于统一调用入口的 API Key 校验

Consumer 仍然是后端内部必要模型，但对终端用户保持隐式，不作为显式业务流程暴露。

### 3.5 Unified Access

负责统一调用入口。

运行方式：

1. 调用方使用 API Key 访问 Hub 统一地址
2. Hub 完成 API Key 校验
3. Hub 解析目标 API 资产
4. Hub 直接向上游地址发起转发请求
5. Hub 将上游响应原样返回给调用方
6. Hub 记录调用日志

需要明确的是：

- Unified Access 当前由 Hub 自行完成转发
- 上游响应不使用 TML Result 包装
- TML Result 仅用于本项目自身业务接口

### 3.6 Observability

负责最小可用调用日志能力。

核心职责：

- 记录统一调用链路中的请求结果
- 支撑当前用户在控制台查询自己的调用日志
- 为后续统计、限额、计费保留扩展空间

## 4. 关键边界

### 4.1 资产边界

API Catalog 负责“资产是什么、归谁、是否可被市场看到、是否可被调用”。

不负责：

- 当前用户登录体系
- API Key 明文签发策略
- 调用日志查询视图
- 具体 HTTP 转发实现细节

### 4.2 发现边界

市场发现只读取已发布资产，不参与资产写入，也不泄露写模型内部配置。

### 4.3 调用边界

Unified Access 只消费 Catalog 给出的可调用资产结论：

- `PUBLISHED`
- `is_deleted = false`
- 上游配置完整

它不维护资产所有权规则。

### 4.4 控制台边界

资产工作台、API Key、日志查询等控制台接口都必须基于当前登录用户上下文，不允许出现跨用户的全局管理写接口。

## 5. 运行时主链路

### 5.1 用户发布 API 资产

1. 用户登录控制台
2. 用户在资产工作台创建 API 资产草稿
3. 用户补充分类、上游配置、示例、AI 元数据
4. 系统校验资产完整性
5. 用户发布资产
6. 资产进入市场可见与可调用集合

### 5.2 用户浏览并调用 API

1. 用户在 API 市场浏览已发布资产
2. 用户创建自己的 API Key
3. 用户携带 API Key 访问统一调用地址
4. Unified Access 校验凭证并解析目标资产
5. Hub 直接转发至上游并返回结果
6. Observability 记录调用日志

### 5.3 用户查看控制台数据

1. 用户进入开发者控制台
2. 查看自己的 API 资产工作台
3. 查看自己的 API Key
4. 查看自己的 API 调用与日志信息

## 6. 第一阶段架构结论

第一期必须落稳的不是“平台录入 API”，而是“用户在市场中发布和使用 API”的完整主链路。

因此本期后端架构的核心结论是：

- Aether API Hub 是用户赋能型 API 市场，不是平台代运营目录
- API 资产必须建模为用户拥有的市场资产
- 市场只展示已发布资产
- Unified Access 只调用已发布资产
- 控制台接口必须严格基于当前用户上下文
- 当前没有独立管理端，只有 API 市场与开发者控制台
